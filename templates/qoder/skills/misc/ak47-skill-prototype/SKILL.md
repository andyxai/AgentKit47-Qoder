# 一次性原型（Prototype）

原型是**用来回答问题的可抛弃代码**。问题决定原型的形态。

## 选择分支

识别要回答的问题——从用户的提示、周围的代码，或者询问用户：

- **"这个逻辑/状态模型感觉对吗？"** → [逻辑原型](#逻辑原型-logic-prototype)。构建一个小型交互式终端应用，将状态机推入那些在纸上难以推理的案例中。
- **"这应该长什么样？"** → [UI 原型](#ui-原型-ui-prototype)。在单个路由上生成几个截然不同的 UI 变体，通过 URL 参数和浮动底部栏切换。

这两个分支产生的产物非常不同——选错会浪费整个原型。如果问题确实模糊且用户不可达，默认选择更符合周围代码的分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部声明假设。

## 适用于两个分支的规则

1. **从第一天就是可抛弃的，并明确标记。** 将原型代码放在实际使用位置附近（紧邻要原型化的模块或页面），以便上下文清晰——但命名要让 casual reader 能看出这是原型，不是生产代码。对于一次性 UI 路由，遵守项目现有的路由约定；不要发明新的顶级结构。

2. **一条命令运行。** 使用项目现有的任务运行器支持的任何方式——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能够不加思考地启动它。

3. **默认无持久化。** 状态存在于内存中。持久化是原型要*检查*的东西，而不是它应该依赖的东西。如果问题明确涉及数据库，使用临时数据库或本地文件，并清楚地命名为"PROTOTYPE — 可删除"。

4. **跳过打磨。** 没有测试，没有超出使原型*可运行*范围的错误处理，没有抽象。重点是快速学习一些东西然后删除它。

5. **展示状态。** 在每次操作后（逻辑）或每次变体切换时（UI），打印或渲染完整的相关状态，以便用户看到变化。

6. **完成后删除或吸收。** 当原型回答了问题后，要么删除它，要么将验证过的决策折叠到真实代码中——不要让它腐烂在仓库里。

## 完成后

原型唯一值得保留的是*答案*。将其捕获到持久的地方（commit message、ADR、issue，或原型旁边的 `NOTES.md`），连同它要回答的问题一起。如果用户在身边，那捕获是一次快速对话；如果不在，留下占位符以便他们（或你在下一轮）在删除原型前填写结论。

## 逻辑原型（Logic Prototype）

一个小型交互式终端应用，让用户手动驱动状态模型。当问题是关于**业务逻辑、状态转换或数据结构**时使用——那种在纸上看起来合理但一旦推入真实案例就感觉不对的东西。

### 何时这是正确的形态

- "我不确定这个状态机是否处理了 X 然后 Y 的边界情况。"
- "这个数据模型真的能让我表示...的情况吗？"
- "我想在写 API 之前感受一下它应该长什么样。"
- 任何用户想要**按按钮并观察状态变化**的情况。

如果问题是"这应该长什么样"——选错分支了。使用 [UI 原型](#ui-原型-ui-prototype)。

### 流程

#### 1. 声明问题

在写代码之前，写下你要原型化的状态模型和问题。一段话，放在原型的 README 或文件顶部注释中。回答错误问题的逻辑原型是纯粹的浪费——明确问题以便稍后检查，无论用户现在是否在看着或 AFK 返回。

#### 2. 选择语言

使用宿主项目使用的任何语言。如果项目没有明显的运行时（例如文档仓库），询问。

匹配项目现有的工具约定——不要仅为原型添加新的包管理器或运行时。

#### 3. 将逻辑隔离到可移植模块

将实际逻辑——回答问题的那部分——放在一个小型、纯接口后面，这个接口以后可以被提取并放入真实代码库。围绕它的 TUI 是可抛弃的；逻辑模块不应该是。

正确的形态取决于问题：

- **纯 reducer** — `(state, action) => state`。当动作是离散事件且状态是单个值时适用。
- **状态机** — 显式的状态和转换。当"现在哪些动作甚至是合法的"是问题的一部分时适用。
- **一组纯函数** 操作普通数据类型。当没有隐式的当前状态——只有转换时适用。
- **具有清晰方法表面的类或模块**，当逻辑确实拥有持续的内部状态时适用。

选择最符合所问问题的形态，*而不是*最容易连接到 TUI 的形态。保持纯净：没有 I/O，没有终端代码，没有用于控制流的 `console.log`。TUI 导入它并调用它；没有任何东西反向流动。

这使得原型在自己的生命周期之外也有用。当问题被回答后，验证过的 reducer/机器/函数集可以被提升到真实模块中——TUI 外壳被删除。

#### 4. 构建最小的 TUI 来暴露状态

将其构建为**轻量级 TUI**——在每个 tick 上，清屏（`console.clear()` / `print("\033[2J\033[H")` / 等效）并重新渲染整个帧。用户应该始终看到一个稳定的视图，而不是不断增长的滚动回溯。

每帧有两个部分，按此顺序：

1. **当前状态**，美观打印且 diff 友好（每行一个字段，或格式化的 JSON）。字段名或节标题使用**粗体**，不重要的上下文（时间戳、ID、派生值）使用**暗色**。原生 ANSI 转义码没问题——`\x1b[1m` 粗体，`\x1b[2m` 暗色，`\x1b[0m` 重置。除非项目中已经有样式库，否则不需要引入。

2. **键盘快捷键**，列在底部：`[a] 添加用户  [d] 删除用户  [t] 时钟滴答  [q] 退出`。粗体键，暗色描述，或反之——只要能清晰阅读。

行为：

1. **初始化状态** — 单个内存中对象/结构。启动时渲染第一帧。
2. **每次读取一个按键（或一行）**，分派到修改状态的处理程序。
3. **每次操作后重新渲染**完整帧——不要追加，替换。
4. **循环直到退出。**

整个帧应该适合一屏。

#### 5. 使其可一条命令运行

添加脚本到项目现有的任务运行器（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应该运行 `pnpm run <prototype-name>` 或等效命令——永远不需要记住路径。

如果宿主项目没有任务运行器，只需将命令放在原型 README 的顶部。

#### 6. 交接

给用户运行命令。他们会自己驱动它；有趣的时刻是他们说"等等，那不应该可能"或"嗯，我假设 X 会不同"——这些是_想法_中的 bug，这正是重点。如果他们想添加新动作，添加它们。原型会演进。

#### 7. 捕获答案

当原型完成工作后，问题的答案是唯一值得保留的东西。如果用户在身边，问它教会了他们什么。如果不在，在原型旁边留下 `NOTES.md` 以便答案可以被填写（或由你填写，如果你观看了会话），然后删除原型。

### 反模式

- **不要添加测试。** 需要测试的原型不再是原型。
- **不要连接到真实数据库。** 使用内存存储，除非问题明确关于持久化。
- **不要泛化。** 没有"如果我们以后想支持 X 会怎样"。原型回答一个问题。
- **不要把逻辑和 TUI 混在一起。** 如果 reducer/状态机引用 `console.log`、提示或终端转义码，它就不再可移植。保持 TUI 作为纯模块的薄外壳。
- **不要将 TUI 外壳发布到生产环境。** 外壳是为了从终端手动驱动而优化的。它背后的逻辑模块才是值得保留的部分。

## UI 原型（UI Prototype）

生成**几个截然不同的 UI 变体**在单个路由上，从浮动底部栏切换。用户在浏览器中翻转变体，选择一个（或从每个中偷取一些），然后扔掉其余的。

如果问题是关于逻辑/状态而不是看起来应该是什么样——选错分支了。使用 [逻辑原型](#逻辑原型-logic-prototype)。

### 何时这是正确的形态

- "这个页面应该长什么样？"
- "我想在承诺之前看看这个仪表板的几个选项。"
- "为设置屏幕尝试不同的布局。"
- 任何用户否则会在脑海中三个模糊模型之间花一天时间挑选的情况。

### 两个子形态——强烈偏好子形态 A

当 UI 原型**与应用的其余部分相邻**时更容易判断——真实头部、真实侧边栏、真实数据、真实密度。单独的一次性路由是真空：每个变体在隔离中看起来都不错。只要有合理的现有页面来托管变体，就默认子形态 A。仅在原型确实没有附近的主页时才使用子形态 B。

#### 子形态 A — 调整现有页面（首选）

路由已存在。变体在**同一路由**上渲染，由 `?variant=` URL 搜索参数门控。现有的数据获取、参数和认证都保留——只有渲染交换。这是默认值；除非有特定理由不选择，否则选择它。

如果原型是针对还没有页面但*自然会存在于其中一个*的东西（仪表板的新部分、设置屏幕上的新卡片、现有流程中的新步骤）——那仍然是子形态 A。在宿主页面内挂载变体。

#### 子形态 B — 新页面（最后手段）

仅当被原型化的东西确实没有现有页面可以放入时才使用——例如一个全新的顶级表面，或无法嵌入任何合理地方的流程。

创建一个**一次性路由**，遵循项目现有的路由约定——不要发明新的顶级结构。命名使其显然是原型（例如在路径或文件名中包含 `prototype` 一词）。相同的 `?variant=` 模式。

在承诺子形态 B 之前，健全性检查：真的没有现有页面可以嵌入吗？空路由会隐藏填充路由会暴露的设计问题。

在两个子形态中，浮动底部栏是相同的。

### 流程

#### 1. 声明问题并选择 N

默认**3 个变体**。超过 5 个就不再是截然不同，开始变成噪音——上限在那里。

用一行写下计划，在原型的位置或文件顶部注释中：

> "设置页面的三个变体，通过 `?variant=` 切换，在现有的 `/settings` 路由上。"

无论用户在这里推回与否，这都有效。

#### 2. 生成截然不同的变体

起草每个变体。将每个变体保持到：

- 页面的目的和它可以访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、纯 CSS，无论什么）。
- 清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须**结构上不同**——不同的布局、不同的信息层次、不同的主要功能，不只是不同的颜色。三个稍微调整的卡片网格不是 UI 原型，它是墙纸。如果两个草稿出来太相似，用明确的"不要使用卡片网格"指导重做一个。

#### 3. 将它们连接在一起

在路由上创建单个切换器组件：

```tsx
// 伪代码 — 适配项目的框架
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：保持切换器上方的所有现有数据获取；只有每个变体的渲染子树变化。

对于子形态 B（新页面）：`/prototype/<name>` 下的一次性路由挂载相同的切换器。

#### 4. 构建浮动切换器

屏幕底部中心的小型固定位置栏，有三部分：

- **左箭头** — 循环到前一个变体（环绕）。
- **变体标签** — 显示当前变体键，如果变体导出名称，也显示该名称。例如 `B — 侧边栏布局`。
- **右箭头** — 向前循环（环绕）。

行为：

- 点击箭头更新 URL 搜索参数（使用框架的路由器——Next 上的 `router.replace`、React Router 上的 `navigate` 等），以便变体可共享且重新加载稳定。
- 键盘：`←` 和 `→` 箭头键也循环。当 `<input>`、`<textarea>` 或 `[contenteditable]` 聚焦时不要拦截箭头键。
- 视觉上与页面不同（例如高对比度药丸、微妙阴影），以便它显然不是正在评估的设计的一部分。
- 在生产构建中隐藏——门控在 `process.env.NODE_ENV !== 'production'` 或等效检查上，以便原型合并不会将栏发送给用户。

将切换器放在单个共享组件中，以便两个子形态都可以重用它。将其定位在项目中共享 UI 所在的任何地方。

#### 5. 交接

显示 URL（和 `?variant=` 键）。用户会在到达时翻转。有趣的反馈通常是**"我想要 B 的头部和 C 的侧边栏"**——那是他们真正想要的设计。

#### 6. 捕获答案并清理

一旦变体获胜，写下哪个和为什么（commit message、ADR、issue，或如果 AFK 运行且用户尚未响应，原型旁边的 `NOTES.md`）。然后：

- **子形态 A** — 删除失败的变体和切换器；将获胜者折叠到现有页面中。
- **子形态 B** — 将获胜变体提升为真实路由，删除一次性路由和切换器。

不要留下变体组件或切换器。它们腐烂很快并混淆下一个读者。

### 反模式

- **仅在颜色或文案上不同的变体。** 那是调整，不是原型。真正的变体在结构上不同意。
- **在变体之间共享太多代码。** 共享 `<Header>` 没问题；共享 `<Layout>` 违背了重点。每个变体应该自由地扔掉布局。
- **将变体连接到真实突变。** 只读原型没问题。如果变体需要突变，指向存根——问题是"这应该长什么样"，而不是"后端是否工作"。
- **直接将原型提升到生产环境。** 变体代码是在原型约束下编写的（没有测试，最小错误处理）。折叠时正确重写它。

## 与 OpenSpec 的关系

### 核心定位

原型是 OpenSpec 变更的**可选技术验证工具**，在 `openspec propose` 之后、`openspec apply-change` 之前使用。

**关键原则**：
- ✅ 原型是**可选的**，不是强制步骤
- ✅ 原型验证 OpenSpec 的设计决策是否正确
- ✅ 原型结论反馈到 OpenSpec 文档，提升质量
- ✅ 原型代码不进入生产，不进入 OpenSpec 归档

### 何时使用原型（验证 OpenSpec）

**推荐使用原型的场景**：
- ✅ 复杂状态机/业务逻辑（design.md 中的架构设计）
- ✅ 多种 UI 方案选择（design.md 中的界面设计）
- ✅ 新业务领域探索（specs/ 中的需求可行性）
- ✅ 审核提出技术风险（critical-review 后的验证）

**可以跳过原型的场景**：
- ❌ 简单 CRUD 操作
- ❌ 成熟的设计模式
- ❌ 需求很明确，风险低
- ❌ 小的配置修改

### 集成流程

```
openspec propose "变更名称"
  ↓
  生成 OpenSpec 文档：
  ├── proposal.md    ← 为什么做
  ├── design.md      ← 怎么做（架构设计）
  ├── specs/         ← 需求规范
  └── tasks.md       ← 实施任务
  ↓
⭐ 原型验证（可选，30 分钟 - 2 小时）
  ├─ 验证 design.md 的架构设计
  ├─ 验证 specs/ 的需求可行性
  └─ 验证 tasks.md 的任务分解合理性
  ↓
  输出：NOTES.md（验证结论）
  ↓
基于原型结论更新 OpenSpec
  ├─ 修改 design.md（调整架构）
  ├─ 修改 specs/（调整需求）
  └─ 修改 tasks.md（调整任务）
  ↓
openspec apply-change "变更名称"
  ↓
  按照验证过的设计实施
```

### 原型验证 OpenSpec 的具体内容

#### 1. 验证 `design.md`

- 状态机设计是否正确？
- 架构决策是否可行？
- 技术选型是否合适？
- 并发处理策略是否充分？

**示例**：
```markdown
<!-- design.md 中的设计 -->
订单状态机：
- pending → paid → shipped → complete
- paid → cancelled

<!-- 原型验证后发现 -->
问题：paid → cancelled 缺少退款流程
决策：更新为 paid → refunded → cancelled
```

#### 2. 验证 `specs/*.md`

- 需求是否可实现？
- 边界情况是否覆盖？
- 性能要求是否合理？
- 并发需求是否可行？

**示例**：
```markdown
<!-- specs/order/spec.md 中的需求 -->
系统必须支持同一订单的并发操作

<!-- 原型验证后发现 -->
发现：需要分布式锁
决策：补充实现要求到 spec.md
```

#### 3. 验证 `tasks.md`

- 任务分解是否合理？
- 任务估算是否准确？
- 依赖关系是否正确？
- 是否有遗漏的子任务？

**示例**：
```markdown
<!-- tasks.md 中的任务 -->
- [ ] Task 1: 实现订单状态机

<!-- 原型验证后发现 -->
发现：状态机比预期复杂，需要拆分
决策：拆分为 3 个子任务
```

### 原型对 OpenSpec 的影响

| OpenSpec 文件 | 原型验证内容 | 原型影响 |
|--------------|-------------|---------|
| **proposal.md** | 范围是否合理 | 调整范围（如果需要） |
| **design.md** | 架构是否正确 | 修改设计决策 |
| **specs/*.md** | 需求是否可行 | 调整需求或补充约束 |
| **tasks.md** | 任务分解是否合理 | 细化或合并任务 |

### 关键原则

1. **原型不改变 OpenSpec 的结构，只提升质量**
   - OpenSpec 文档结构保持不变
   - 原型结论更新文档内容

2. **原型的结论必须反馈到 OpenSpec 文档**
   - 记录到 NOTES.md
   - 更新 design.md/specs/tasks.md
   - 确保实施基于验证过的设计

3. **原型代码不进入 OpenSpec 归档**
   - `openspec archive-change` 时删除原型代码
   - 保留重要的 NOTES.md 到经验文档

4. **原型是可选的，但复杂变更强烈建议使用**
   - 简单变更：跳过原型，节省时间
   - 复杂变更：使用原型，避免返工
