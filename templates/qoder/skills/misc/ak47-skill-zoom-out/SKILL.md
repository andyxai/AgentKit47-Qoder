# 放大视角（Zoom Out）

**我不熟悉这个代码区域。请提升一层抽象。给我所有相关模块和调用者的地图，使用项目的领域词汇。**

## 何时使用

- 当你不熟悉某段代码时
- 当你需要理解代码如何融入更大图景时
- 当你需要查看模块间的关系时
- 当你需要识别依赖关系时

## 触发方式

用户可以说：
- "放大看一下这个模块"
- "这个代码在整个系统中处于什么位置？"
- "给我这个区域的上下文地图"
- "我不熟悉这段代码，请提升抽象层次"

或者 AI 主动触发：
- 当探索新代码区域时
- 当需要理解模块依赖时
- 当准备进行跨模块修改时

## 输出格式

AI 必须提供：

```markdown
# {模块名称} 上下文地图

## 模块定位
{这个模块在系统中的位置和职责}

## 上层抽象
{这个模块属于哪个更大的组件/系统}

## 相关模块
| 模块 | 关系 | 说明 |
|------|------|------|
| {模块 A} | 依赖/被依赖/平级 | {如何交互} |
| {模块 B} | 依赖/被依赖/平级 | {如何交互} |

## 调用者
- {谁调用这个模块}
- {在什么场景下调用}

## 被调用者
- {这个模块调用什么}
- {为什么调用}

## 数据流
{数据如何流入和流出这个模块}

## 关键概念
{使用项目领域词汇解释核心概念}
```

## 使用项目领域词汇

**必须**使用 `CONTEXT.md` 中定义的术语：
- 检查 `.qoder/CONTEXT.md` 是否存在
- 如果存在，使用其中定义的术语
- 如果不存在，使用通用术语并建议创建 CONTEXT.md

## 示例

### 示例 1: 查看订单模块

**用户**: "放大看一下订单处理模块"

**AI**:
```markdown
# Order Processing 上下文地图

## 模块定位
Order Processing 负责处理订单的生命周期，从创建到完成或取消。

## 上层抽象
属于 Order Management 系统的一部分，与 Inventory Management 和 Payment Processing 平级。

## 相关模块
| 模块 | 关系 | 说明 |
|------|------|------|
| Order Model | 被依赖 | 订单数据模型 |
| Inventory Service | 依赖 | 检查库存可用性 |
| Payment Gateway | 依赖 | 处理支付 |
| Notification Service | 被依赖 | 发送订单状态通知 |

## 调用者
- API Layer - HTTP 请求触发订单操作
- Admin Dashboard - 管理员手动操作订单
- Scheduled Jobs - 超时订单自动取消

## 被调用者
- Inventory API - 预留/释放库存
- Payment API - 扣款/退款
- Email Service - 发送订单确认

## 数据流
客户端请求 → API → Order Processing → Inventory/Payment → 更新状态 → 通知

## 关键概念
- Order: 客户的购买请求
- OrderItem: 订单中的单个商品
- OrderStatus: 订单当前状态（pending/paid/shipped/cancelled）
```
