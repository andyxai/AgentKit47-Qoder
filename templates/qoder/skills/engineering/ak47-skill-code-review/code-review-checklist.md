# 代码审查清单（Code Review Checklist）

> **用途**：ak47-agent-reviewer 执行代码审查时的检查清单  
> **两层机制**：第一层快速检查（必做）+ 第二层深度检查（按需）  
> **参考**：[重构最佳实践](../../../../.ak47/experiences/best-practices/BP-refactoring-improving-existing-code.md)

---

## 第一层：快速检查清单（必做）

**目标**：5-10 分钟内完成，捕获明显问题。

### P0 - 阻塞性问题（必须修复）

| # | 检查项 | 检查方法 | 通过标准 |
|---|--------|---------|---------|
| 1 | **功能完整性** | 对比需求/Spec | 所有需求点已实现 |
| 2 | **测试覆盖** | 运行测试套件 | 所有测试通过，核心逻辑有测试 |
| 3 | **安全漏洞** | 检查输入验证、权限校验 | 无注入风险，敏感数据已保护 |
| 4 | **明显 Bug** | 检查逻辑、边界条件 | 无逻辑错误，边界条件已处理 |

**快速检查方法：**
```bash
# 运行测试
npm test

# 检查变更范围
git diff --stat {BASE_SHA}..{HEAD_SHA}

# 检查测试覆盖率
npm run test:coverage
```

**判断标准：**
- ❌ 任何一项未通过 → **Critical**，必须修复
- ✅ 全部通过 → 继续 P1 检查

---

### P1 - 重要问题（应该修复）

| # | 检查项 | 检查方法 | 通过标准 | 触发第二层阈值 |
|---|--------|---------|---------|---------------|
| 1 | **错误处理** | 检查异常处理、错误返回 | 异常已妥善捕获和处理 | - |
| 2 | **类型安全** | 检查类型定义、类型转换 | 类型准确，无隐式转换 | - |
| 3 | **命名清晰** | 检查函数/变量名 | 名称表达“做什么”而非“怎么做” | - |
| 4 | **代码重复** | 检查重复代码块 | 无明显复制粘贴（三次法则） | 重复 > 3 处 |
| 5 | **函数长度** | 检查函数行数 | 函数 ≤ 50 行（特殊情况除外） | 函数 > 100 行 |
| 6 | **嵌套条件** | 检查 if-else 嵌套层数 | 嵌套 ≤ 2 层 | 嵌套 > 3 层 |
| 7 | **参数数量** | 检查函数参数个数 | 参数 ≤ 4 个 | 参数 > 6 个 |

**快速检查方法：**
```bash
# 查找重复代码（人工检查）
grep -n "重复的代码片段" src/**/*.ts

# 查找过长函数
# 人工审查 > 50 行的函数

# 查找魔法数字
grep -n "[0-9]\{2,\}" src/**/*.ts | grep -v "const\|//\|*"
```

**判断标准：**
- ⚠️ 发现 1-2 个问题 → **Important**，应该修复
- ⚠️ 发现 > 3 个问题 → 记录所有问题，继续 P2 检查
- ⚠️ 达到“触发第二层阈值” → 标记线索，在审查报告中建议深度检查
- ✅ 无问题或少量可接受问题 → 继续 P2 检查

---

### P2 - 改进建议（记录后续处理）

| # | 检查项 | 检查方法 | 通过标准 | 触发第二层阈值 |
|---|--------|---------|---------|---------------|
| 1 | **性能隐患** | 检查循环、查询、资源使用 | 无明显性能问题 | - |
| 2 | **复杂度** | 检查嵌套、条件分支 | switch ≤ 5 分支 | switch > 8 分支 |
| 3 | **类长度** | 检查类的总行数 | 类 ≤ 200 行 | 类 > 300 行 |
| 4 | **文档注释** | 检查复杂逻辑的注释 | 复杂逻辑有清晰注释 | - |
| 5 | **代码风格** | 检查格式、一致性 | 符合项目风格指南 | - |
| 6 | **接口设计** | 检查 get/set 方法比例 | 有业务行为，非纯数据类 | get/set > 10 个 |
| 7 | **调用链长度** | 检查对象链式调用 | 链式 ≤ 2 层 | 链式 > 3 层 |

**判断标准：**
- 💡 发现问题 → **Minor**，记录到技术债清单
- ✅ 无问题 → 审查完成

---

## 何时升级到第二层深度检查

**触发条件（满足任一即触发）：**

| 线索 | 触发阈值 | P1 关注阈值 | 可能的问题 | 第二层检查路径 |
|------|---------|------------|-----------|---------------|
| 重复代码 | > 3 处 | > 2 处 | 违反 DRY 原则 | 路径1：重复代码检查 |
| 嵌套条件 | > 3 层 | > 2 层 | 箭头型代码 | 路径2：条件逻辑检查 |
| 函数长度 | > 100 行 | > 50 行 | 职责不单一 | 路径3：函数/类职责检查 |
| 类长度 | > 300 行 | > 200 行 | 过大类 | 路径3：函数/类职责检查 |
| 参数数量 | > 6 个 | > 4 个 | 过长参数列 | 路径4：接口设计检查 |
| switch分支 | > 8 分支 | > 5 分支 | 条件逻辑复杂 | 路径2：条件逻辑检查 |
| get/set方法 | > 10 个 | > 5 个 | 贫血模型 | 路径4：接口设计检查 |
| 调用链长度 | > 3 层 | > 2 层 | 消息链耦合 | 路径5：耦合依赖检查 |

**执行策略：**
- **P1 关注阈值**：在 P1/P2 检查时标记为“需要关注”，记录到审查报告
- **触发第二层阈值**：在审查报告中明确建议“触发深度检查”，说明理由
- **审查者判断**：最终是否执行第二层由审查者根据上下文决定

## 第二层：深度检查清单（按需）

**目标**：当第一层发现严重线索时，深入分析代码质量。

### 路径1：重复代码检查

**参考**：[重构最佳实践 - 重复代码](../../../../.ak47/experiences/best-practices/BP-refactoring-improving-existing-code.md#1-重复代码)

**检查项：**

| 检查点 | 特征 | 重构手法 |
|--------|------|---------|
| 同一类内重复 | 相同代码块出现 ≥ 2 次 | Extract Method |
| 父子类间重复 | 父子类有相同逻辑 | Pull Up Method |
| 不同类间重复 | 不同类有相同逻辑 | Extract Class / Extract Superclass |
| 条件分支内重复 | 各分支有相同代码 | Consolidate Duplicate Conditional Fragments |

**检查方法：**
```bash
# 查找相似代码块（人工审查）
# 关注：相似的 if-else、相似的循环、相似的数据处理逻辑
```

**评估标准：**
- 🔴 重复代码导致维护困难 → 必须重构
- 🟡 重复代码但影响有限 → 建议重构
- 🟢 重复是合理的（如配置文件） → 接受

---

### 路径2：条件逻辑检查

**参考**：[重构最佳实践 - 条件逻辑复杂](../../../../.ak47/experiences/best-practices/BP-refactoring-improving-existing-code.md#大类四条件逻辑复杂问题)

**检查项：**

| 检查点 | 特征 | 重构手法 |
|--------|------|---------|
| 嵌套条件 > 3 层 | 箭头型代码 | Replace Nested Conditional with Guard Clauses |
| switch/case > 5 分支 | switch 惊悚 | Replace Conditional with Polymorphism |
| 重复条件表达式 | 同一条件多处出现 | Consolidate Conditional Expression + Extract Method |
| 控制标记变量 | found/done 等标记 | Remove Control Flag |

**检查方法：**
```typescript
// 坏例子：嵌套条件
if (user) {
  if (user.isActive) {
    if (user.hasPermission) {
      // 业务逻辑
    }
  }
}

// 好例子：卫语句
if (!user) return;
if (!user.isActive) return;
if (!user.hasPermission) return;
// 业务逻辑
```

**评估标准：**
- 🔴 嵌套深、难理解 → 必须重构
- 🟡 条件多但逻辑清晰 → 建议优化
- 🟢 条件简单、可读性好 → 接受

---

### 路径3：函数/类职责检查

**参考**：[重构最佳实践 - 代码臃肿与冗余](../../../../.ak47/experiences/best-practices/BP-refactoring-improving-existing-code.md#大类一代码臃肿与冗余类问题)

**检查项：**

| 检查点 | 特征 | 重构手法 |
|--------|------|---------|
| 过长函数 > 50 行 | 承担多个职责 | Extract Method |
| 过大类 > 300 行 | 违背单一职责 | Extract Class |
| 临时字段 | 只在特定场景使用 | Extract Class |
| 过度设计 | 未使用的抽象/扩展点 | Inline Class/Method |

**检查方法：**
```bash
# 查找大文件
find src -name "*.ts" -size +10k

# 查找长函数（人工审查）
# 关注：包含多个注释块的函数、职责不清晰的函数
```

**评估标准：**
- 🔴 职责混乱、难以维护 → 必须重构
- 🟡 职责较多但有逻辑 → 建议拆分
- 🟢 职责单一、清晰 → 接受

---

### 路径4：接口设计检查

**参考**：[重构最佳实践 - 函数与接口设计缺陷](../../../../.ak47/experiences/best-practices/BP-refactoring-improving-existing-code.md#大类五函数与接口设计缺陷问题)

**检查项：**

| 检查点 | 特征 | 重构手法 |
|--------|------|---------|
| 过长参数列 > 5 个 | 调用困难 | Introduce Parameter Object |
| 贫血模型 | 只有 get/set | Move Method（形成充血模型） |
| 查询修改混合 | 函数有副作用 | Separate Query from Modifier |
| 错误码代替异常 | return -1 表示失败 | Replace Error Code with Exception |

**检查方法：**
```typescript
// 坏例子：过长参数列
function createUser(name: string, email: string, age: number, 
                    address: string, phone: string, role: string) {}

// 好例子：参数对象
interface CreateUserParams {
  name: string;
  email: string;
  age: number;
  address: string;
  phone: string;
  role: string;
}
function createUser(params: CreateUserParams) {}
```

**评估标准：**
- 🔴 接口难用、易出错 → 必须重构
- 🟡 接口可用但不优雅 → 建议优化
- 🟢 接口清晰、易用 → 接受

---

### 路径5：耦合依赖检查

**参考**：[重构最佳实践 - 耦合与依赖过度](../../../../.ak47/experiences/best-practices/BP-refactoring-improving-existing-code.md#大类三耦合与依赖过度问题)

**检查项：**

| 检查点 | 特征 | 重构手法 |
|--------|------|---------|
| 发散式变化 | 一个类因多原因被修改 | Extract Class |
| 霰弹式修改 | 一个需求改多个类 | Move Method + Move Field |
| 依恋情结 | 函数大量操作其他类数据 | Move Method |
| 消息链过长 | a.getB().getC().getD() | Hide Delegate |
| 中间人 | 类的方法只是委托 | Remove Middle Man |

**检查方法：**
```bash
# 查找长调用链
grep -n "\.get[A-Z].*\.get[A-Z].*\.get[A-Z]" src/**/*.ts

# 查找循环依赖（人工审查）
# 关注：A 引用 B，B 又引用 A
```

**评估标准：**
- 🔴 强耦合、难以修改 → 必须重构
- 🟡 有耦合但有边界 → 建议优化
- 🟢 松耦合、依赖清晰 → 接受

## 审查原则

### DO ✅
- 按 P0 → P1 → P2 顺序检查
- 每个问题提供具体位置（file:line）
- 说明为什么问题重要
- 提供修复建议（如果不明显）
- 承认优点

### DON'T ❌
- 不说"看起来不错"而不检查
- 不把细节问题标记为 Critical
- 不对没审查的代码给反馈
- 不给出模糊的反馈（"改进错误处理"）
- 避免不给明确结论
