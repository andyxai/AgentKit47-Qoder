# 垂直切片任务分解（Vertical Slicing）

使用垂直切片（追踪子弹）将计划分解为可独立抓取的切片。

## 核心概念

### 垂直切片 vs 水平切片

**垂直切片（✅ 正确）**：
- 穿过所有层的薄切片（schema → API → UI → tests）
- 每个切片可独立演示
- 窄但完整的路径

**水平切片（❌ 错误）**：
- 只做一层（"先做所有 API"、"再做所有 UI"）
- 无法独立演示
- 集成风险高

### HITL vs AFK 切片

- **HITL**（Human-In-The-Loop）：需要人工交互，如架构决策或设计审查
- **AFK**（Away From Keyboard）：可全自动实施和合并，无需人工交互
- **优先 AFK**：尽可能选择 AFK 切片

## 流程

### 1. 收集上下文

从对话上下文中工作。如果用户传递了需求文档引用（文件路径或内容），读取完整内容。

### 2. 探索代码库（可选）

如果尚未探索代码库，进行探索以理解代码当前状态。切片标题和描述应使用项目的领域词汇，并尊重相关区域的 ADR。

### 3. 起草垂直切片

将计划分解为**追踪子弹**切片。每个切片是一个薄的垂直切片，端到端穿过**所有**集成层，**不是**某一层的水平切片。

**垂直切片规则**：
- 每个切片通过每层提供窄但**完整**的路径（schema、API、UI、tests）
- 完成的切片可独立演示或验证
- 偏好多个薄切片而非少数厚切片

**切片设计原则**：

1. **端到端行为**：描述端到端行为，不是逐层实现
2. **独立可验证**：每个切片完成后可以独立测试
3. **依赖明确**：清楚标识切片间的依赖关系
4. **用户价值**：每个切片应该交付一部分用户价值

**示例**：

❌ **水平切片（错误）**：
```
切片 1: 创建数据库 schema
切片 2: 实现 API 端点
切片 3: 构建 UI 组件
切片 4: 编写测试
```

✅ **垂直切片（正确）**：
```
切片 1: 用户可以创建订单（完整路径：schema → API → UI → tests）
切片 2: 用户可以查看订单列表
切片 3: 用户可以取消订单
切片 4: 管理员可以批量处理订单
```

### 4. 向用户展示并确认

将提议的分解显示为编号列表。对每个切片，显示：

- **标题**: 简短描述性名称
- **类型**: HITL / AFK
- **阻塞于**: 哪些其他切片（如果有）必须先完成
- **覆盖的用户故事**: 这解决了哪些用户故事（如果源材料有）

**询问用户**：

- 粒度感觉对吗？（太粗 / 太细）
- 依赖关系正确吗？
- 任何切片应该合并或进一步拆分吗？
- 正确的切片标记为 HITL 和 AFK 了吗？

迭代直到用户批准分解。

### 5. 生成切片文档

对于每个批准的切片，生成切片文档。使用下面的切片模板。这些切片被认为已准备好实施，所以按依赖顺序生成（阻塞者先行）。

**切片模板**：

```markdown
# 切片 {N}: {标题}

**类型**: AFK | HITL  
**阻塞于**: {阻塞的切片编号，或"无 - 可以立即开始"}  
**覆盖的用户故事**: {用户故事描述}

## 构建什么

这个垂直切片的简洁描述。描述端到端行为，不是逐层实现。

避免具体的文件路径或代码片段——它们很快过时。例外：如果原型产生了比散文更精确地编码决策的片段（状态机、reducer、schema、类型形状），在此内联并简短注明它来自原型。修剪到决策丰富的部分——不是工作演示，只是重要的部分。

## 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 涉及的技术层

- **Schema**: {数据库/schema 变更}
- **API**: {API 端点/变更}
- **UI**: {UI 组件/变更}
- **Tests**: {测试范围}

## 边界情况

- {边界情况 1 及处理策略}
- {边界情况 2 及处理策略}

## 参考

- 相关需求文档: {链接}
- 相关 ADR: {链接}
- 原型输出: {如果有}
```

### 6. 切片执行顺序

按依赖顺序执行切片：

1. **识别阻塞关系**: 哪些切片必须先完成
2. **排序**: 生成执行顺序（无依赖的先行）
3. **并行机会**: 识别可以并行执行的切片
4. **里程碑**: 每个切片完成是一个里程碑

**示例执行顺序**：

```
切片 1: 用户可以创建订单（无依赖）
  ↓
切片 2: 用户可以查看订单列表（依赖切片 1）
切片 3: 用户可以取消订单（依赖切片 1）
  ↓（切片 2 和 3 可并行）
切片 4: 管理员可以批量处理订单（依赖切片 2 和 3）
```

## 反模式

### 反模式 1: 水平切片

**错误**：
```
"我们先做后端，再做前端"
```

**正确**：
```
"我们先做一个完整的用户故事，从数据库到 UI"
```

---

### 反模式 2: 切片太厚

**错误**：
```
"这个切片实现所有订单管理功能"
```

**正确**：
```
"这个切片只实现创建订单，其他功能后续切片"
```

---

### 反模式 3: 切片无法独立演示

**错误**：
```
"这个切片只做数据库 schema"
```

**正确**：
```
"这个切片实现完整的创建订单流程，可以演示"
```

---

### 反模式 4: 依赖关系不明确

**错误**：
```
不标识切片间的依赖
```

**正确**：
```
明确标识：切片 2 依赖切片 1
```

## 与其他 Skill 的关系

| Skill | 关系 | 说明 |
|-------|------|------|
| **openspec-propose** | 前置 | 切片基于 OpenSpec 的 tasks.md |
| **prototype** | 可选前置 | 原型验证 OpenSpec 后进行切片（可选） |
| **subagent-driven-development** | 后续 | 切片可以分发给子 Agent 并行 |
| **executing-plans** | 后续 | 切片作为计划的一部分执行 |

## 红线

- ❌ 创建水平切片（只做一层）
- ❌ 切片无法独立演示
- ❌ 依赖关系不明确
- ❌ 切片太厚（应该拆分为多个薄切片）
- ❌ 不经过用户确认就生成切片

---

**最终状态是生成批准的切片文档，按依赖顺序执行。** 每个切片是一个独立的、可演示的、端到端的用户价值交付。
