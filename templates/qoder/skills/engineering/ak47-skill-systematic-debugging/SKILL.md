# Systematic Debugging - 6 阶段系统化调试循环

> **基于 mattpocock diagnose Skill 设计**
> 
> 完整实现了 6 阶段调试循环，包括：
> - Phase 1: 10 种反馈循环构建方法
> - Phase 2: Bug 复现验证
> - Phase 3: 3-5 个可证伪假设
> - Phase 4: 仪器化（一次一个变量）
> - Phase 5: 修复 + 回归测试
> - Phase 6: 清理 + 事后分析 + 架构改进交接

## 核心原则

> **NO FIX WITHOUT UNDERSTANDING ROOT CAUSE**
> 
> 没有理解根因之前，绝不修复。

违反这个流程的字面意思，就是违反调试的精神。

## 何时使用

**任何技术问题时都必须使用**：
- 测试失败
- 运行时错误
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别是在以下情况**：
- ⏰ 时间紧迫（紧急情况下更容易猜测）
- 🔥 "看起来很简单"的问题（简单问题也有根因）
- 🔄 已经尝试了多次修复
- ❌ 之前的修复没起作用
- 🤔 你不完全理解问题

## 6 阶段调试循环

**必须按顺序完成每个阶段，除非有明确的理由跳过。**

当探索代码库时，使用项目的领域术语表（CONTEXT.md）获得清晰的模块心智模型，并检查你涉及区域的 ADRs。

### Phase 1 — 构建反馈循环（核心！）

**这是这个 Skill 的核心。** 其他都是机械操作。

如果你有一个快速、确定性、可自动运行的 pass/fail 信号来检测 Bug，你一定能找到根因——二分法、假设测试、仪器化都只是消费这个信号。

**如果没有反馈循环，再多的盯代码也救不了你。**

#### 在这里投入不成比例的努力

> **Spend disproportionate effort here.**
> 
> **要激进。要有创造性。拒绝放弃。**

#### 10 种构建方法（按优先级尝试）

1. **失败的测试用例**：在能触及 Bug 的任何层面（单元测试、集成测试、e2e）
2. **curl / HTTP 脚本**：对着运行中的开发服务器
3. **CLI 调用**：使用 fixture 输入，将 stdout 与已知正确的快照对比
4. **无头浏览器脚本**（Playwright / Puppeteer）：驱动 UI，断言 DOM/console/network
5. **回放捕获的 trace**：保存真实的网络请求/payload/事件日志到磁盘；在隔离环境中重放到代码路径
6. **一次性 harness**：启动系统的最小 Subset（一个服务，mock 依赖），用单次函数调用练习 Bug 代码路径
7. **Property / fuzz 循环**：如果 Bug 是"有时输出错误"，运行 1000 个随机输入寻找失败模式
8. **Bisection harness**：如果 Bug 出现在两个已知状态之间（commit、数据集、版本），自动化"在状态 X 启动，检查，重复"，以便 `git bisect run`
9. **Differential 循环**：将相同输入通过旧版本 vs 新版本（或两个配置）运行，diff 输出
10. **HITL bash 脚本**（最后手段）：如果必须人工点击，用 `scripts/hitl-loop.template.sh` 驱动他们，使循环仍然结构化

**构建正确的反馈循环，Bug 就 90% 修复了。**

#### 迭代优化反馈循环本身

把反馈循环当作产品。一旦你有了_一个_循环，问：

- ⚡ **能更快吗？**（缓存 setup，跳过不相关的 init，缩小测试范围）
- 🎯 **能更精确吗？**（断言具体症状，而不是"没崩溃"）
- 🔒 **能更确定性吗？**（固定时间、seed RNG、隔离文件系统、冻结网络）

**一个 30 秒的 flaky 循环几乎和没有循环一样差。一个 2 秒的确定性循环是调试超能力。**

#### 非确定性 Bug

目标不是干净的复现，而是**更高的复现率**。

循环触发 100 次，并行化，添加压力，缩小时间窗口，注入 sleeps。

**50% 的 flaky Bug 是可调试的；1% 不是**——持续提升复现率直到可调试。

#### 多组件系统诊断（AK47 增强）

**当系统有多个组件时（CI → build → signing，API → service → database）**：

在提出修复方案之前，**添加诊断 instrumentation**：

```
对于 EACH 组件边界：
  - 记录什么数据进入组件
  - 记录什么数据离开组件
  - 验证环境/config 传播
  - 检查每层状态

运行一次收集证据，显示 WHERE 它失败
然后分析证据识别失败的组件
然后调查那个特定组件
```

**示例（多层系统）**：
```bash
# Layer 1: Workflow
echo "=== Secrets available in workflow: ==="
echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

# Layer 2: Build script
echo "=== Env vars in build script: ==="
env | grep IDENTITY || echo "IDENTITY not in environment"

# Layer 3: Signing script
echo "=== Keychain state: ==="
security list-keychains
security find-identity -v

# Layer 4: Actual signing
codesign --sign "$IDENTITY" --verbose=4 "$APP"
```

**这揭示了**：哪层失败（secrets → workflow ✓, workflow → build ✗）

#### 当你真的无法构建循环时

**停下来，明确说明。**

列出你尝试了什么。请求用户提供：
- (a) 能复现的环境访问
- (b) 捕获的 artifact（HAR 文件、日志 dump、core dump、带时间戳的录屏）
- (c) 添加临时生产 instrumentation 的权限

**没有反馈循环，不要进入假设阶段。**

**✅ 通过条件**：你相信这个反馈循环，才能进入 Phase 2

### Phase 2 — 复现 Bug

运行循环。看着 Bug 出现。

确认：

- [ ] 循环产生的是**用户描述**的失败模式——不是附近的其他失败。**错误的 Bug = 错误的修复**
- [ ] 失败在多次运行中可复现（或对于非确定性 Bug，以足够高的比率复现）
- [ ] 你捕获了精确的症状（错误信息、错误输出、慢时间），以便后续阶段可以验证修复是否真的解决了它

**✅ 通过条件**：你复现了 Bug，才能进入 Phase 3

---

### Phase 3 — 生成假设

在测试任何假设之前，生成 **3-5 个排序的假设**。

**单假设生成会锚定在第一个看似合理的想法上。**

每个假设必须是**可证伪的**：陈述它做出的预测。

> **格式**："如果 <X> 是原因，那么 <改变 Y> 会使 Bug 消失 / <改变 Z> 会使它更糟"

**如果你无法陈述预测，这个假设只是一个感觉——丢弃或锐化它。**

#### 示例

| # | 假设 | 验证方法 | 可证伪性 |
|---|------|---------|---------|
| 1 | 如果问题是**缓存过期**，那么清除缓存后 Bug 应该消失 | 清除缓存，运行测试 | ✅ 如果 Bug 仍在，假设被证伪 |
| 2 | 如果问题是**权限不足**，那么提升权限后 Bug 应该消失 | 修改权限配置 | ✅ 如果 Bug 仍在，假设被证伪 |
| 3 | 如果问题是**并发竞态**，那么添加锁后 Bug 应该消失 | 添加 Mutex 锁 | ✅ 如果 Bug 仍在，假设被证伪 |

#### Cheap Checkpoint（低成本检查点）

**在测试之前，向用户展示排序列表。**

他们通常有领域知识可以立即重新排序（"我们刚刚部署了 #3 的变更"），或者知道他们已经排除的假设。

**低成本检查点，巨大时间节省。不要阻塞——如果用户 AFK，按你的排序继续。**

**✅ 通过条件**：有 3-5 个假设（用户确认或 AFK），才能进入 Phase 4

---

### Phase 4 — 仪器化（Instrumentation）

每个 probe 必须映射到 Phase 3 的一个特定预测。

**一次只改变一个变量。**

#### 工具偏好

1. **调试器 / REPL 检查**（如果环境支持）。**一个断点胜过十个日志。**
2. **针对性日志**：在区分假设的边界处。
3. ❌ **绝不**"log everything and grep"。

#### 调试日志规范

**用唯一前缀标记每个调试日志**，例如 `[DEBUG-a4f2]`。

清理时变成单个 grep。**有标记的日志死亡；无标记的日志存活。**

```typescript
console.log('[DEBUG-a4f2] Phase 4: Testing hypothesis 3');
console.log('[DEBUG-a4f2] Token before refresh:', token);
console.log('[DEBUG-a4f2] Token after refresh:', newToken);
```

#### 性能分支

对于性能回归，日志通常是错的。

**改为**：建立基线测量（timing harness、`performance.now()`、profiler、query plan），然后二分法。

**先测量，后修复。**

#### 数据流追溯（AK47 增强）

**当错误在调用栈深处时**：

参考 `root-cause-tracing.md`（本目录）获取完整的反向追溯技术。

**快速版本**：
- 坏值从哪里来？
- 什么用坏值调用了这个？
- 持续向上追溯直到找到源头
- 在源头修复，不是在症状处修复

**✅ 通过条件**：假设被验证/证伪，才能进入 Phase 5

---

### Phase 5 — 修复 + 回归测试

**在修复之前**写回归测试——但仅当存在**正确的测试缝**时。

**正确的测试缝**是测试在调用点练习**真实 Bug 模式**的地方。

如果唯一可用的缝太浅（单调用者测试当 Bug 需要多个调用者，无法复现触发 Bug 的链的单元测试），那里的回归测试给出虚假信心。

#### 如果不存在正确的测试缝

**这本身就是发现。**

记录下来。**代码库架构阻止了 Bug 被锁定。**

标记这个，用于下一阶段。

#### 如果存在正确的测试缝

1. 将最小化复现转化为该缝的失败测试
2. 看着它失败
3. 应用修复
4. 看着它通过
5. 对着原始（未最小化）场景重新运行 Phase 1 反馈循环

#### 防御性编程（AK47 增强）

修复根因后，考虑是否需要在其他层添加防御：

- 输入验证（zod/runtime validation）
- 空值检查
- 边界检查
- 错误处理

参考 `defense-in-depth.md`（本目录）获取完整指南。

**✅ 通过条件**：回归测试通过（或无测试缝已记录），才能进入 Phase 6

---

### Phase 6 — 清理 + 事后分析

**在声明完成之前，必须完成**：

- [ ] 原始复现不再复现（重新运行 Phase 1 循环）
- [ ] 回归测试通过（或无测试缝已记录）
- [ ] **所有 `[DEBUG-...]` instrumentation 已删除**（`grep` 前缀）
- [ ] 一次性原型已删除（或移动到清晰标记的 debug 位置）
- [ ] **正确的假设在 commit / PR message 中陈述**——以便下一个调试者学习

**Commit message 格式**：
```
fix(scope): 修复 XXX Bug

根因：{Phase 3 中正确的假设}
调试过程：{简述 Phase 1-5 的关键步骤}
预防措施：{Phase 6 的预防措施}
```

#### 然后问：什么能预防这个 Bug？

| 预防措施 | 实施方式 | 适用场景 |
|---------|---------|---------|
| **添加测试** | 回归测试入库 | 所有 Bug |
| **添加类型检查** | zod/runtime validation | 数据类型问题 |
| **添加防御性代码** | 空值检查、边界检查 | 边界情况 |
| **添加监控** | 错误追踪、告警 | 生产环境问题 |
| **架构改进** | 模块解耦、接口抽象 | 测试困难的问题 |

#### 架构问题识别（AK47 增强）

**如果 3+ 次修复尝试失败，质疑架构**：

**表明架构问题的模式**：
- 每次修复都揭示新的共享状态/耦合/其他地方的问题
- 修复需要"大规模重构"才能实现
- 每次修复都在其他地方创建新症状

**停下来质疑基础**：
- 这个模式根本上合理吗？
- 我们是否"纯粹因为惯性而坚持"？
- 我们应该重构架构而不是继续修复症状吗？

**在尝试更多修复之前，与你的人类伙伴讨论**

**这不是失败的假设——这是错误的架构。**

如果答案涉及架构变更（无好的测试缝、纠缠的调用者、隐藏耦合），将具体细节交给 `/improve-codebase-architecture` Skill。

**在修复完成后提出建议，而不是在之前**——你现在比开始时拥有更多信息。

**✅ 通过条件**：所有清理完成 + 知识沉淀，才能声明完成

## 🚩 红旗警告 - 停下来遵循流程

如果你发现自己在想：
- "先快速修复，稍后调查"
- "试着改 X 看看是否有效"
- "添加多个变更，运行测试"
- "跳过测试，我手动验证"
- "可能是 X，让我修复那个"
- "我不完全理解但这可能有效"
- "模式说 X 但我会不同地适配"
- "主要问题是：[列出修复而没有调查]"
- 在追溯数据流之前提出解决方案
- **"再试一次修复"（当已经尝试 2+ 次时）**
- **每次修复都在不同地方揭示新问题**

**所有这些都意味着：停下来。回到 Phase 1。**

**如果 3+ 次修复失败**：质疑架构（见 Phase 6）

---

## 👤 你的伙伴表明你做错的信号

**注意这些重定向**：
- "不是那样吗？" - 你没有验证就假设
- "会显示给我们...吗？" - 你应该添加证据收集
- "停止猜测" - 你在没有理解的情况下提出修复
- "深入思考这个" - 质疑基础，不只是症状
- "我们卡住了？"（沮丧） - 你的方法不起作用

**看到这些时**：停下来。回到 Phase 1。

## 常见合理化借口

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 Bug 很快。 |
| "紧急，没时间走流程" | 系统化调试比猜测-检查反复**更快**。 |
| "先试这个，然后调查" | 第一次修复设定模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未测试的修复不持久。先测试证明它。 |
| "一次多个修复省时间" | 无法隔离什么有效。导致新 Bug。 |
| "参考太长，我会适配模式" | 部分理解保证 Bug。完整阅读。 |
| "我看到问题了，让我修复" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再次修复。 |

## 当流程揭示"无根因"时

如果系统化调查揭示问题真的是环境依赖、时间依赖或外部因素：

1. 你完成了流程
2. 记录你调查了什么
3. 实现适当的处理（retry、timeout、错误信息）
4. 添加监控/日志用于未来调查

**但是**：95% 的"无根因"案例是调查不完整。

## 真实世界影响

从调试会话：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时反复
- 首次修复率：95% vs 40%
- 引入的新 Bug：接近零 vs 常见
