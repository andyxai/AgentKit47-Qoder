# 反模式检查 Skill

## 用途

编写代码、审查代码、设计架构或编写 Spec 时，加载本 Skill 对照检查，及时发现并纠正常见陷阱。

## 设计反模式

| 反模式 | 特征 | 修复建议 |
|--------|------|---------|
| 过度工程 | 为简单问题引入复杂架构 | 先MVP验证，再渐进增强 |
| 抽象泄漏 | 底层细节渗透到上层接口 | 封装边界清晰，接口语义完整 |
| 过早优化 | 未验证瓶颈就优化 | 先测量，后优化，数据驱动 |
| 万能类 | 一个类承担过多职责 | 按单一职责拆分 |
| 循环依赖 | 模块A依赖B，B又依赖A | 引入中间层或接口解耦 |

## 实现反模式

| 反模式 | 特征 | 修复建议 |
|--------|------|---------|
| 魔法数字 | 代码中硬编码无意义常量 | 提取为命名常量或配置项 |
| 重复代码 | 相同逻辑散落在多处 | 提取函数/组件/工具方法 |
| 深层嵌套 | 缩进超过4层的条件链 | 提前返回、卫语句、策略模式 |
| 空 catch | 捕获异常后不处理 | 至少记录日志，明确处理策略 |
| 长参数列表 | 函数参数超过5个 | 封装为配置对象或上下文 |

## 流程反模式

| 反模式 | 特征 | 修复建议 |
|--------|------|---------||
| 跳过评审 | 以"改动小"为由绕过审查 | 所有代码变更必须审查 |
| 忽略测试 | 不写测试或测试覆盖虚高 | 测试行为，不测试实现 |
| 文档漂移 | 代码改了，文档/Spec没更新 | 变更流程强制同步文档 |
| 需求镀金 | 实现超出需求范围的功能 | 严格按Spec范围开发 |
| **批量更新遗漏** | 路径/名称/API变更时只改部分文件 | **修改前全局搜索所有引用点，修改后再次搜索验证** |

## 批量更新完整性检查清单

当执行以下操作时，必须使用此检查清单：
- 目录重命名/移动
- 文件路径变更
- API/接口名称修改
- 配置项重命名
- 依赖版本升级

### 检查步骤

1. **修改前：全局搜索**
   ```bash
   # 搜索所有引用点
   grep -r "旧名称" --include="*.md" --include="*.ts" --include="*.js" --include="*.json" --include="*.yaml" .
   ```

2. **修改中：记录清单**
   - 列出所有需要修改的文件
   - 按文件类型分组（文档/代码/配置）
   - 逐个修改并标记

3. **修改后：再次搜索验证**
   ```bash
   # 确认无残留引用
   grep -r "旧名称" --include="*.md" --include="*.ts" --include="*.js" --include="*.json" --include="*.yaml" .
   # 应该返回 0 结果
   ```

4. **提交前：最终确认**
   - 检查 git status 是否包含所有修改的文件
   - 确认没有遗漏的配置/文档文件

### 常见遗漏点

- ❌ 设计文档中的路径引用
- ❌ Skill/Agent README 中的来源说明
- ❌ 配置文件中的忽略规则
- ❌ 示例代码中的导入路径
- ❌ 历史版本记录/CHANGELOG

## 使用方式

1. **编码时**: 对照“实现反模式”逐条自查
2. **审查时**: 对照三类反模式列表标记问题
3. **设计时**: 重点检查“设计反模式”和“流程反模式”
4. **批量修改时**: 执行“批量更新完整性检查清单”

## 输出格式

```yaml
check_result:
  issues_found: 3
  issues:
    - type: "设计反模式"
      name: "过度工程"
      location: "src/core/orchestrator.ts"
      suggestion: "简化状态机，先用if-else验证逻辑"
    - type: "实现反模式"
      name: "魔法数字"
      location: "src/utils/config.ts:42"
      suggestion: "提取为 MAX_RETRY_COUNT 常量"
```

## 红线

- ❌ 发现反模式不记录 = 未检查
- ❌ 标记为"后续修复"却不创建跟踪项 = 无限拖延
