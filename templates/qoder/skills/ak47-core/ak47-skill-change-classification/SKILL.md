# 变更分类 Skill

## 用途

entry-guard 判定为"重量修改"后，加载本 Skill 执行范式分级，将变更映射到 L1/L2/L3 流程。

## 判定决策树

```
收到重量修改请求
    │
    ├─ Q1: 用户可见的需求发生了变化？
    │       ├─ 是 → L1（需求驱动变更）
    │       └─ 否 → 继续
    │
    ├─ Q2: 技术方案/架构设计发生了变化？
    │       ├─ 是 → L2（技术实现变更）
    │       └─ 否 → 继续
    │
    └─ Q3: 纯代码修复/优化，需求和方案均未变？
            ├─ 是 → L3（缺陷修复/代码优化）
            └─ 否 → 上报主Agent人工判定
```

## 判定信号速查表

| 判定信号 | 典型表述 | 范式 |
|---------|---------|------|
| 需求变更 | "加一个新功能" / "用户流程要改" / "UI调整" | L1 |
| 技术实现 | "改用微服务" / "重构模块" / "引入缓存层" | L2 |
| 缺陷修复 | "修复Bug" / "解决内存泄漏" / "优化性能" | L3 |
| 边界模糊 | 同时涉及需求+技术方案变化 | 取最高级 → L1 |

## 输出格式

```yaml
paradigm: "L1" | "L2" | "L3"
confidence: "high" | "medium" | "low"
reasoning: "判定理由（一句话说明依据哪个信号）"
next_action: "启动 L1流程 / 启动 L2流程 / 启动 L3流程"
```

## 关键红线

- ❌ 绝不可将需求变更判定为 L2/L3（低估影响）
- ❌ 绝不可将纯代码修复判定为 L1/L2（过度流程）
- ❌ 禁止以"改动小"为由降级判定

## 与 entry-guard 的分工

| | entry-guard | change-classification |
|---|---|---|
| 职责 | 判定是不是修改 | 把修改分到 L1/L2/L3 |
| 时机 | 任何输入的第一步 | entry-guard 判定为"重量修改"后 |
| 输出 | 答疑 / 轻量 / 重量 | L1 / L2 / L3 |

两者前后衔接，互不替代。
