# 改进审核（Improvement Audit）

## 用途

对 `ak47-skill-improvement-proposal` 产出的改进提案进行独立审核，判断每条建议是否应该被接纳，以及如何落地。

审核的核心是交叉验证：既要从 AK47 设计原则角度审视修改是否合理，也要从用户项目角度确认问题是否真实存在、建议是否对症。

## 触发条件

- 改进提案已完成，需要独立审核
- 用户要求对改进建议"把关"
- AK47 项目维护者需要对提案做接纳决策

## 前置条件

审核者必须完整阅读以下内容：

### AK47 侧（理解工具的设计意图和约束）

| 文件 | 目的 |
|------|------|
| `AGENTS.md` | AI 行为指令和设计哲学 |
| `templates/qoder/rules/*.md` | 规则体系的完整定义 |
| `docs/design/architecture/01-system-overview.md` | 系统架构概览 |
| `docs/design/decisions/` | 已有架构决策记录 |
| `PROJECT-INTRO.md` | 项目介绍和核心定位 |
| `.ak47/config.yaml` | 当前项目配置 |

### 用户项目侧（理解用户的具体诉求）

- 改进提案中的"问题背景"和"用户项目上下文"
- 用户项目的配置和使用场景
- 问题发生时的具体流程

### 提案本身

- 完整的改进提案报告
- 每条建议的根因分析和象限分类

## 审核流程

### 第一步：理解提案

1. 读取完整的改进提案
2. 对每个缺陷和建议建立理解：
   - 说的是什么问题？
   - 根因是什么？
   - 建议怎么修？

### 第二步：AK47 侧交叉验证

对每条建议，从 AK47 设计视角回答：

1. **修改范围边界检查**（硬规则，优先于其他维度）：

   | ✅ AK47 自身配置 | ❌ 外部安装的配置 |
   |---|---|
   | `templates/qoder/rules/*.md` | `openspec-propose/SKILL.md` |
   | `templates/qoder/skills/ak47-core/**/*.md` | `openspec-apply-change/SKILL.md` |
   | `templates/qoder/skills/engineering/**/*.md` | `openspec-archive-change/SKILL.md` |
   | `templates/qoder/skills/productivity/**/*.md` | `openspec-explore/SKILL.md` |
   | `templates/qoder/hooks/*.sh` | `superpowers-*/SKILL.md` |
   | `AGENTS.md`, `CONTEXT.md` | Qoder 平台 settings.json |
   | `.ak47/*.yaml` | OpenSpec CLI 配置文件 |
   | `src/` 目录下的 AK47 源码 | `node_modules/` |
   | `templates/` 目录下的其他模板 | `.qoder/` 下非 AK47 创建的 Skill |

   **裁决规则**：如果建议涉及修改外部安装的配置 → 裁决为 ↗️转上游 或 ❌拒绝（说明需由上游项目实现）。

2. **设计一致性检查**：这个修改是否符合 AK47 的设计哲学？
   - 参考 `AGENTS.md` 的核心原则
   - 参考已有架构决策记录
   - 如果与历史决策冲突，标注出来

2. **影响面分析**：
   - 修改会影响哪些模块？（模板、规则、Skill、CLI）
   - 是否影响已部署的用户项目？（向后兼容性）
   - 是否与外部工具职责重叠？（openspec、Qoder 平台）

3. **实施可行性**：
   - 修改难度评估
   - 是否有更简单的替代方案？
   - 是否可以分阶段实施？

### 第三步：用户项目侧交叉验证

从用户项目视角回答：

1. **必要性检查**：这个问题在用户项目中真实存在吗？
   - 是"确实阻碍了工作"还是"可以接受的摩擦"？
   - 有没有更轻量的解决方案？

2. **象限校正**：提案中的象限分类是否准确？
   - 象限 I（AK47 配置缺陷）→ 容易修复，评估是否正确？
   - 象限 II（AK47 设计缺陷）→ 评估是否需要设计讨论？
   - 象限 III（用户配置问题）→ 是否需要指导而非修改代码？
   - 象限 IV（外部工具限制）→ 是否需要提给上游？

3. **问题类型校正**：提案中标注的问题类型是否准确？

   | 提案标注 | 检查要点 | 常见误判 |
   |----------|----------|----------|
   | 🔵 需求层增加 | AK47 是否真的缺少该能力？还是已有但未被触发？ | 将"规则没生效"误判为"缺少规则" |
   | 🟠 架构设计层修改 | 现有结构是否真的有问题？还是执行偏差导致的表象？ | 将"AI 没按流程走"误判为"流程设计错了" |
   | 🟢 执行偏差 | 规则是否足够明确？触发条件是否可操作？ | 将"规则本身不合理"误判为"执行力度不够" |

   **校正输出**：对每条建议标注审核后的问题类型，如有调整说明原因。

### 第四步：综合裁决

对每条建议给出裁决：

| 裁决 | 含义 | 后续动作 |
|------|------|----------|
| ✅ 接纳 | 建议合理，纳入实施 | 创建对应 task |
| 🔄 修改后接纳 | 方向对但方案需调整 | 提出修改意见 |
| ⏸️ 暂缓 | 有价值但当前优先级低 | 标记 backlog |
| ❌ 拒绝 | 不符合设计原则或非普适问题 | 说明拒绝理由 |
| ↗️ 转上游 | 需要外部工具支持 | 标注目标上游项目 |

### 第五步：产出审核报告

```markdown
# 改进提案审核报告

> **审核人**: AI（加载 ak47-skill-improvement-audit）
> **审核日期**: YYYY-MM-DD
> **审核对象**: {提案标题/路径}

## 一、审核概要

- 提案共 {N} 条建议
- 接纳 {X} 条 / 修改后接纳 {Y} 条 / 暂缓 {Z} 条 / 拒绝 {W} 条
- 转上游 {V} 条

## 二、逐条审核

### 建议 1: {建议标题}

- **提案问题类型**: {提案标注的类型}
- **审核后问题类型**: {校正后的类型，如有调整说明原因}
- **提案象限**: {提案中的分类}
- **审核后象限**: {校正后分类，如有调整说明原因}
- **裁决**: {✅接纳 / 🔄修改后接纳 / ⏸️暂缓 / ❌拒绝 / ↗️转上游}
- **AK47 视角**:
  - 设计一致性: {符合/部分符合/不符合}，原因: ...
  - 影响面: {涉及哪些模块}
  - 实施难度: {高/中/低}
- **用户项目视角**:
  - 必要性: {真实痛点/可接受摩擦/过度要求}
  - 替代方案: {有没有更轻量的方式？}
- **裁决理由**: {综合两方视角的判断}

### 建议 2: ...
（逐条重复）

## 三、最终修改方案

### 接纳项（含修改后接纳）

对每条接纳的建议，给出可直接执行的修改方案：

| # | 建议 | 问题类型 | 涉及文件 | 修改内容 | 优先级 |
|---|------|----------|----------|----------|--------|
| 1 | {建议标题} | 🔵/🟠/🟢 | `path/to/file` | {具体怎么改：新增某章节、替换某段、删除某行} | P0 |

### 暂存 backlog

| # | 建议 | 暂缓原因 | 重新评估时机 |
|---|------|----------|--------------|
| 1 | ... | {为什么现在不做} | {什么条件下重新考虑} |

### 拒绝项

每条拒绝必须给出明确理由，禁止模糊表述：

| # | 建议 | 拒绝理由 | 建议替代方向 |
|---|------|----------|--------------|
| 1 | {建议标题} | {具体原因：非普适问题/与设计原则冲突/外部工具职责/...} | {如果问题确实存在，建议怎么解决} |

### 转上游

| # | 建议 | 目标项目 | 问题描述 |
|---|------|----------|----------|
| 1 | ... | openspec/Qoder | {需要上游支持什么} |

## 四、审核声明

"本审核从 AK47 设计原则和用户项目实际诉求两个视角完成交叉审核。
所有裁决均附有明确理由。"
```

- ❌ 所有建议一刀切拒绝（防御型审核）
- ❌ 裁决不给理由
- ❌ 将外部工具能力边界内的需求标注为 AK47 缺陷
