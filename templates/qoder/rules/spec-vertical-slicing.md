# Spec 垂直切片规则

> **注入方式**: 需在 Qoder IDE **Settings → Rules** 中配置类型后，Qoder 平台自动注入。核心摘要已内联至 `AGENTS.md`，本文件为完整版可选增强。
> **定位**: 强制约束 OpenSpec spec 的切分方式，禁止水平切分。
> **优先级**: 硬规则（每次创建或评审 specs 时必须遵守）。

---

## 核心原则

**每个 Spec 必须描述一个端到端可独立演示和验证的用户价值路径，禁止按技术组件/层次水平切分。**

---

## 强制要求

### 1. 垂直切片定义

- 必须：Spec 按"端到端用户价值"垂直切分：从输入到输出，穿越所有技术层
- 必须：每个 spec 描述一个可独立编译、测试、验证的完整行为路径
- 必须：每个 spec 有一个明确的"演示场景"：用户做什么 → 系统返回什么

### 2. 禁止水平切分

- 禁止：按技术组件单独建 spec（如 Fetcher、Extractor、Classifier）
- 禁止：按技术层次单独建 spec（如 API 层、Service 层、DAO 层）
- 禁止：按模块/包名单独建 spec（如 pipeline-core、entity-matching）

---

## 垂直切片示例

### 正确：按用户价值切分

| Spec | 用户场景 | 端到端路径 |
|------|---------|-----------|
| V1: 基础站点类型判定 | 输入 URL → 返回站点类型 | URL → Fetch → Extract → Classify → 类型结果 |
| V2: 影视站点完整分类 | 输入影视 URL → 返回完整分类 | URL → Fetch → Extract → 合规检查 → 实体匹配 → 多维分类 → 结果 |
| V3: 批量站点分类 | 输入 URL 列表 → 返回批量结果 | URLs → 并发 Fetch → 各站点分类 → 聚合结果 |

### 错误：按组件水平切分

| Spec | 问题 |
|------|------|
| H1: Fetcher 实现 | 只做一层，无法独立验证端到端用户价值 |
| H2: Extractor 实现 | 同上 |
| H3: Entity Matcher 实现 | 同上 |

---

## 审查检查点

`ak47-skill-critical-review` 在执行时必须检查以下项目：

- [ ] 每个 Spec 是否描述一个端到端可演示的用户价值？
- [ ] Spec 是否按技术组件/层次水平切分？（若是则为致命问题，必须重新切分）
- [ ] 每个 Spec 是否可独立编译、测试、验证？
- [ ] Spec 之间是否有明确的用户价值差异（而非技术实现差异）？

---

## 与 openspec-propose 的关系

- `openspec-propose` skill 是中立的模板生成工具，不决定切分策略
- 切分策略由本规则约束，在 AI 生成 spec 时生效
- 问题不在 "openspec 命令不行"，而在缺少本约束规则
