# 架构决策演进方法论：从教条到务实

---

## 背景

在 AI 编程的工程实践中，经常会制定各种规则来约束 AI 行为。初始版本的规则通常偏向教条，随着实践深入需要根据实际场景进行调整。

本文记录了一套在实践中总结的五步演进法，帮助你将僵化的教条式规则迭代为灵活的务实方案。一句话：架构决策应服务于目标，而非束缚实践。

---

## 来源

这个方法论起源于 AK47 开发中的一次关键争论：经验总结应该由谁来做？

初始规则（AGENTS.md 中）明确规定：主 Agent 禁止提炼经验，必须由 Knowledge Engineer 子 Agent 执行。这听起来很合理——职责分明。但实际执行中发现：

- 主 Agent 拥有完整的对话上下文（知道问题是怎么讨论出来的）
- 子 Agent 没有上下文（需要重新解释一遍）
- 上下文丢失导致经验总结质量下降
- 用户觉得"为什么不能直接做？"

用户的一句关键提问启动了演进："如果主 Agent 用 Skill 也能实现吧？"

这引发了从"教条禁止"到"场景分流"的五步演进。

---

## 原理

核心洞察：规则的价值不是"禁止了什么"，而是"在什么场景下服务了什么目标"。一刀切的教条式规则看似安全，实际上可能违反了更高优先级的原则（如上下文连续性）。

五个演进阶段解决了不同层次的问题：
1. **教条规则** → 明确了"底线"（但一刀切）
2. **技术分析** → 确认了"可行性"（但没解决"为什么"）
3. **价值发现** → 识别了"什么更重要"（关键转折点）
4. **场景分流** → 给不同场景设计了不同方案
5. **全局审核** → 确保一致性，堵住遗漏

---

## 核心方法：五步演进法

```
教条规则
  │
  ├─ 1. 技术可行性分析（"能否做到？"）
  │     ↓
  ├─ 2. 质疑规则合理性（"为什么禁止？"）
  │     ↓
  ├─ 3. 发现核心价值（"什么更重要？"）
  │     ↓
  ├─ 4. 场景分流设计（"按场景分类"）
  │     ↓
  └─ 5. 全局一致性审核（"检查遗漏"）
        ↓
    务实方案
```

### 阶段 1：教条规则（初始状态）

**特征**：
- 规则明确但僵化
- "必须"/"禁止"类表述过多
- 缺乏场景区分
- 违反规则被视为错误

**典型案例**：
> "AGENTS.md 禁止主 Agent 提炼经验，必须由 KE 子 Agent 执行。"

**问题**：
- 忽略了实际场景的多样性
- 导致效率损失（上下文丢失、协调开销）
- 产生抵触情绪

**这个阶段要问**：这个规则解决了什么问题？

---

### 阶段 2：技术可行性分析

**触发**：用户提问"如果主 Agent 用 Skill 也能实现吧"

**动作**：
- 承认技术上可行
- 分析为什么架构上仍禁止（职责边界）
- 但开始反思：禁止的理由是否充分？

**这个阶段要问**：
- 技术上是否可行？
- 架构上为什么禁止？
- 规则解决了什么问题？代价是什么？

---

### 阶段 3：核心价值发现（关键转折点）

**触发**：用户提出关键洞察
> "主 Agent 因为有上下文，一些比如产品需求讨论编写、代码开发编写，这些上下文会提高效果的都需要用主 Agent 结合 Skill 会更好"

**动作**：
- 识别出"上下文连续性"是核心价值
- 对比"教条职责分工" vs "上下文优先"
- 发现原规则违反了更高优先级的原则

**关键洞察**：
- **上下文连续性 > 职责边界清晰度**
- 效率损失是真实存在的代价
- 规则应为场景服务，而非一刀切

**这个阶段要问**：有没有更高优先级的原则被违反了？

---

### 阶段 4：场景分流设计

**动作**：
- 按场景分类（即时上下文 vs 跨会话）
- 为每类场景设计最优方案
- 明确决策矩阵和判断标准

**输出**：
- 决策流程图
- 场景分类表
- 检查清单

**这个阶段要问**：不同场景是否有不同的最优解？

---

### 阶段 5：全局一致性审核

**触发**：用户要求"你再全局审核下看看是不是还有遗漏"

**动作**：
- 检查相关文档的一致性（如 AGENTS.md 中的相关表格）
- 补充缺失的场景说明
- 细化笼统的规则
- 添加委托规则示例

**这个阶段要问**：相关文档是否一致？是否有遗漏？

---

## 各阶段关键问题速查

| 阶段 | 关键问题 |
|------|---------|
| 1. 教条规则 | 这个规则解决了什么问题？ |
| 2. 技术分析 | 技术上是否可行？架构上为什么禁止？ |
| 3. 价值发现 | 有没有更高优先级的原则被违反？ |
| 4. 场景分流 | 不同场景是否有不同的最优解？ |
| 5. 全局审核 | 相关文档是否一致？是否有遗漏？ |

## 常见陷阱

### ❌ 陷阱 1：过早优化规则

**表现**：在缺乏实际场景验证时，制定过于详细的规则。

**后果**：规则与实际场景不匹配，需要频繁修订。

**修正**：先在实践中积累案例，再提炼规则。

---

### ❌ 陷阱 2：忽视用户反馈

**表现**：用户提出质疑时，坚持原有规则而非反思。

**后果**：用户失去信任，规则执行效果差。

**修正**：将用户质疑视为规则优化的信号，而不是需要"说服"的问题。

---

### ❌ 陷阱 3：忽略隐性成本

**表现**：只看到规则的好处，忽略其代价（效率损失、上下文丢失、认知负担）。

**后果**：规则看似合理，实际执行效果差。

**修正**：评估规则时，同时列出"收益"和"代价"两栏。

---

## 检查清单

### 规则设计检查
- [ ] 规则是否按场景分类？（避免一刀切）
- [ ] 是否识别了核心价值？（什么优先级最高）
- [ ] 是否评估了规则的代价？（效率、复杂度、用户体验）
- [ ] 是否有决策流程图？（便于执行者判断）
- [ ] 是否经过全局一致性审核？（相关文档是否冲突）

### 演进过程检查
- [ ] 是否经历了技术可行性分析？
- [ ] 是否质疑了规则的合理性？
- [ ] 是否发现了更高优先级的原则？
- [ ] 是否按场景进行了分流设计？
- [ ] 是否进行了全局一致性审核？
