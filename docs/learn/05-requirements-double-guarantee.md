# 05 · 需求文档双重保障机制

> **解决什么问题**：需求写完后怎么减少返工？需求和架构混在一起导致流程混乱、作者自己审核自己、验收标准模糊  
> **预计阅读**：10 分钟  
> **原始经验**：[BP-constitution-double-guarantee.md](../../.ak47/experiences/best-practices/BP-constitution-double-guarantee.md)
>
> 📎 灵感来源：[mattpocock/skills](https://github.com/mattpocock/skills) — `/grill` skill 的"一次一问 + HARD-GATE"设计理念

## 背景

一套四阶段门控机制，确保需求和架构设计在进入实施前经过两道独立审核——让问题在文档阶段就暴露出来，而不是在代码写完后才发现需求理解错了。

```
需求定义 → 批判性审核 → 📄 需求文档存档 → 架构设计 → 批判性审核 → 📄 架构文档存档 → 进入实施
   ↑            ↑                            ↑            ↑
  阶段 1       阶段 2                       阶段 3        阶段 4
  (创作)      (独立审核)                    (创作)       (独立审核)
```

核心创新点：**创作和审核分离**——设计者不能审核自己的文档，必须由独立视角进行批判性质疑。

---

## 来源

### 背景问题

在 AK47 的设计中，`requirements-grilling` 这个 Skill 同时承载了**需求澄清**和**架构设计**两个职责：

```
❌ 早期设计（单 Skill 混合模式）
用户提需求 → AI 盘问 → 用户回答 → AI 直接给技术方案 → 用户说"可以" → 开始写代码
```

这带来了四个问题：

| 问题 | 具体表现 |
|------|---------|
| **流程混淆** | 需求盘问完成后没有明确的归档节点，直接跳到技术方案 |
| **缺乏阶段门控** | 需求和架构混在一个连续流程中，没有明确的里程碑 |
| **作者思维定势** | 设计者自己审核自己的文档，难以发现假设和遗漏 |
| **验收标准模糊** | "用户批准"没有量化标准，依赖主观判断 |

### 用户的关键反馈

> "怎么算是完全达成一致的理解？"
> 
> "是不是能有一个带有批判性思维的 skill，进行批判性审核"

这两句话直接催生了双重保障机制的核心设计：**独立的批判性审核作为第二道防线，以及可量化的验收标准。**

## 原理

### 机制原理

| 保护层 | 做什么 | 防止什么 |
|--------|--------|---------|
| **第一层：需求定义** | 一次一问盘问 + 术语统一 + 反向复述 | 需求理解偏差、术语不一致 |
| **第二层：需求审核** | 6 大维度批判性质疑 + 用户回应 | 遗漏假设、边界不清 |
| **第三层：架构设计** | 基于已审核需求 + 多方案对比 + 分章节验证 | 技术与需求脱节、过度设计 |
| **第四层：架构审核** | 再次 6 大维度批判性质疑 | 设计缺陷、风险低估 |

### 为什么需要独立审核？

作者存在**思维定势**：自己写的东西，隐式假设和逻辑漏洞很难被自己发现——就像自己写的代码不容易审查出自己的 Bug。这类似于代码审查中的基本共识：**另一个人看才能发现你忽略的问题。**

### 为什么需要四要素验收？

主观判断（"用户说可以"）容易遗漏关键问题。四要素验收把批准变成可测量的标准：

| 阶段 | 四要素 | 量化指标 |
|------|--------|---------|
| **需求定义** | 术语零歧义 | 术语覆盖率尽可能达到 100% |
|  | 需求可验证 | 模糊需求数 = 0 |
|  | 反向复述通过 | AI 复述 → 用户确认"正确" |
|  | 显式批准 | 用户明确说"我批准"（拒绝"好的"） |
| **架构设计** | 完整性 | 至少 2 种方案对比 |
|  | 可追溯性 | 每个设计决策都能追溯到需求 |
|  | 风险沟通 | 至少识别 3 个风险 |
|  | 显式批准 | 用户明确说"我批准" |

## 核心方法

### 批判性审核的 6 大维度

审核者从以下 6 个维度对文档进行系统性质疑：

| 维度 | 审核重点 | 示例问题 |
|------|---------|---------|
| **1. 假设识别** | 未明说的隐式假设 | "你假设用户都有高速网络，这对吗？" |
| **2. 边界情况** | 极端场景处理 | "如果并发量是预期的 10 倍会怎样？" |
| **3. 替代方案** | 是否有更好的方案 | "为什么不用事件驱动而不是轮询？" |
| **4. 可维护性** | 长期维护成本 | "这个方案 6 个月后还容易改吗？" |
| **5. 风险评估** | 风险是否被低估 | "你说风险低，但有数据支持吗？" |
| **6. 一致性** | 与现有系统/术语一致 | "这个术语和 CONTEXT.md 定义一致吗？" |

### 质疑清单分级

审核产出按优先级三级分类，不要求解决所有问题，但要求显式记录决策：

```markdown
## 🔴 高优先级（必须解决或用户确认接受风险）
- [ ] 假设：用户都有高速网络 → 需要降级方案
- [ ] 边界：并发 10 倍时性能下降 90% → 需要优化

## 🟡 中优先级（强烈建议解决）
- [ ] 可维护性：模块耦合度高 → 建议解耦
- [ ] 替代方案：可考虑事件驱动 → 评估迁移成本

## 🟢 低优先级（可选优化）
- [ ] 命名：某变量名不够清晰 → 建议重命名
```

### 通过标准

| 优先级 | 要求 |
|--------|------|
| 🔴 高优先级 | 尽量全部解决 **或** 用户明确确认接受风险 |
| 🟡 中优先级 | ≥ 50% 解决 |
| 🟢 低优先级 | 已记录，可后续处理 |

**关键灵活性**：用户可以**显式接受风险**。不是所有问题都必须立即解决——关键是记录决策，而非强制消除所有风险。

## 对比其他方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **双重保障机制（本方案）** | 阶段门控清晰、独立审核、可量化验收、保留风险知情权 | 流程步骤多、初期感觉"慢" | 中大型项目、多人协作、需求复杂的系统 |
| **单 Skill 混合模式** | 流程短、感觉"快" | 需求和技术混淆、无独立审核、验收模糊、容易遗漏 | 不推荐 |
| **瀑布式重文档** | 文档详尽、追溯性强 | 流程僵化、反馈周期长、AI 时代效率低 | 合规要求极高的场景 |
| **敏捷轻文档** | 快速迭代、灵活 | 缺乏阶段门控、需求漂移难控制、技术债累积快 | 小型探索性项目 |

### 为什么不能"简单项目跳过流程"？

一个常见的反模式：「这个功能很简单，不用走完整流程。」问题是：

1. **"简单"是主观判断**——你眼中的简单功能可能涉及你没想到的边界情况
2. **HARD-GATE 的价值恰恰在"你以为简单"的时候**——当你觉得不需要检查时，往往最需要检查
3. **流程不会让简单变复杂**——简单功能走一遍流程很快，但能拦住 80% 的低级遗漏
