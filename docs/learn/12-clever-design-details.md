# 12 精巧设计详解

> AK47 中有不少看似细小、实则关键的设计决策。这些机制单独看都不起眼，合在一起才构成"让 AI 不乱来"的防线。以下逐一展开。

---

## 背景

AI 编程助手的核心问题是**行为不可预测**——它可能在没理解需求时就开始写代码，可能跳过关键步骤自认为"不需要"，可能在提交代码前没有审查，可能假设用户同意了就继续推进。

传统的解决思路是"写更详细的 Prompt"，但 Prompt 越长 AI 越容易遗漏。AK47 的思路不同：**把软约束硬化**——用门控、Hook、日志、格式规范等手段，把"应该"变成"必须"。

---

## 来源

这些机制不是凭空设计，而是在实际使用中踩坑后逐步补上的：

- **门控纪律** 源自多次 AI 写完代码直接提交、跳过审查的教训
- **1% 规则** 源自 AI 多次自判"功能简单不需要 TDD"然后写出没测试的代码
- **复述确认** 源自 AI 在需求阶段理解偏差、到实现阶段才发现问题的返工
- **垂直切片** 源自分 Spec 时按组件切导致多个 Spec 互相依赖、无法独立交付验证
- **偏离日志** 源自"这次特殊跳过吧"但没有记录、后来无法追溯的混乱
- **Hook 警告映射** 源自 AI 看到 Hook 警告后"嗯知道了"然后继续原来操作的教训
- **状态实时推断** 源自缓存状态与实际文件不一致导致的流程错乱
- **阻塞检测** 源自多个 Change 并行时某些 Change 被遗忘、几周后才发现的经历
- **反馈行为准则** 源自 AI 过度讨好导致问题被掩盖、或过度对抗导致协作破裂
- **禁止行为清单** 源自 AI 反复触碰同一类错误的模式识别
- **知识沉淀自动触发** 源自"这次记得记录"变成"下次一定"的循环

每个机制背后都有具体的踩坑经历，提炼成了可复用的规则。

---

## 原理

### 为什么"硬化约束"有效

AI 处理自然语言指令时，倾向于按"概率最高的解释"执行——不一定是"最准确的解释"。当指令是"应该先写测试"时，AI 可能理解为"大多数情况下先写测试，但这个场景特殊"。当指令是"如果没有测试文件就停止编码"时，AI 没有模糊空间。

AK47 的约束硬化策略分三层：

```
软约束（散文式）        →  AI 容易忽略
    ↓
硬规则（AGENTS.md）     →  AI 上下文中明确列出
    ↓
可执行（Hook 脚本）     →  不依赖 AI 理解，直接拦截
```

三层递进，不把鸡蛋放在一个篮子里。

---

## 核心方法

### 🚦 三层门禁架构

```
第一层：Git Hooks（执行层）
  commit-msg 格式、敏感信息检测 → 零 Token，不依赖 AI

第二层：AGENTS.md 规则（上下文层）
  1% 规则、禁止行为、门控纪律 → Qoder 自动注入 AI 上下文

第三层：AI 自律（行为层）
  Hook 警告必须响应、门控必须等人确认、跳过必须写偏离日志
```

**设计要点**：每层独立工作。Hook 失效了规则层还在，规则层被忽略了自律层还在。不假设任何一层一定生效。

### 🚪 11 个门控点 + 范式感知

| # | 门控点 | 问法示例 | L1 | L2 | L3 |
|---|--------|---------|:--:|:--:|:--:|
| G1 | 需求理解确认 | "需求理解是否正确？是否可以进入变更分类？" | ● | ● | ○ |
| G2 | 变更分类确认 | "当前变更判定为 L{X}，对应流程强度为 {强度}，是否确认？" | ● | ● | ● |
| G3 | proposal 评审 | "proposal 已创建，是否通过评审？" | ● | ● | ○ |
| G4 | design 评审 | "design 已创建，是否通过评审？" | ● | ● | ○ |
| G5 | specs 评审 | "specs 是否按垂直切片组织？是否覆盖所有需求？" | ● | ● | ○ |
| G6 | tasks 评审 | "tasks 拆分是否合理？粒度是否合适？" | ● | ● | ● |
| G7 | 批判性审核 | "是否确认进入实施阶段？" | ● | ● | ○ |
| G8 | 代码审查 | "是否启动代码审查？" | ● | ● | ● |
| G9 | 审查通过 | "审查通过，是否提交代码？" | ● | ● | ● |
| G10 | 功能验收 | "是否启动验收测试？" | ● | ● | ● |
| G11 | 分支处理 | "是否合并分支/创建 PR？" | ● | ● | ● |

> ● = 必须执行 ○ = 可选/简化

**范式感知**：不是所有变更都需要完整 11 个门控。Bug 修复（L3）跳过需求确认、proposal、design、specs；架构重构（L1）全量执行。门控强度跟随变更风险动态调整。

**确认格式规范**：AI 必须用明确询问句式（"是否通过？"），不能假设用户同意（"没问题的话我就继续了"），一次只等一个门控点的回复，不能合并询问。

### 🔴 1% 规则 + 跳过判定

**1% 规则**："如果有 1% 可能某个 Skill 适用，你必须调用它。"

这不是修辞手法，是操作指令。配套的「跳过判定规则」关上了后门：

> AI 认为当前场景不需要某个 Skill → 必须 1) 明确说明原因 → 2) 询问用户是否同意跳过 → 3) 用户同意后才能跳过

同时有 5 个「严禁自判跳过」的 Skill 白名单：entry-guard、change-classification、triage-brief、TDD、vertical-slicing——AI 对这几个 Skill **无权**自行判断"不需要"。

**为什么是 1% 而不是 100%**：因为 AI 天然倾向于走捷径。如果说"100% 确定才调用"，AI 永远能找到理由说"这个场景只有 90%"。1% 是个不可谈判的阈值。

### 🔄 复述确认理解

需求盘问完成后，AI 必须做三件事：

1. **用自己的话复述**对需求的理解（不是复制粘贴用户原话）
2. **明确提出确认请求**（"以上理解是否正确？"）
3. **等待用户明确回复**（"对"/"需要修改第 X 点"）

**防止 AI 自说自话**：AI 的常见行为是"我觉得我理解了"就开始写代码。复述确认强制 AI 把理解显式化，给用户纠偏的机会。这对应门控体系中的 G1。

### 📐 Spec 垂直切片

**规则**：每个 Spec 必须描述一个端到端可独立演示和验证的用户价值路径，禁止按技术组件/层次水平切分。

```
✅ 正确："用户上传图片并查看压缩结果"（端到端，穿越上传→压缩→存储→展示）
❌ 错误："Fetcher 模块 Spec"、"API 层 Spec"（按组件/层次水平切分）
```

**为什么严格**：水平切分导致多个 Spec 互相依赖，无法独立开发、独立测试、独立演示。critical-review 执行时会把违反此规则标记为 🔴 致命问题。

### 📝 偏离日志

任何门控点如需跳过，必须四步记录：

1. **说明跳过原因**（仅限紧急修复/用户明确要求）
2. **明确告知用户**跳过了哪个门控点
3. **记录到** `.ak47/deviations.log`
4. **获得用户明确批准**后方可跳过

日志格式记录了时间、门控点、原因、批准人，所有"偷懒"可追溯。

### 🖥️ Git Hooks 零 Token 消耗

commit-msg 格式检查、敏感信息检测等硬规则由 Git Hook 用 Shell 脚本处理，`git commit` 时自动触发：

- 不经过 AI 上下文 → 不消耗 Token
- 不依赖 AI 理解 → 不会因为"上下文太长没读到"而失效
- 与 AGENTS.md 规则形成互补——Hook 拦截语法层错误，规则约束行为层问题

### 🧠 自我改进闭环

当 AK47 自身行为不符合预期：

1. **改进提案 Skill**：深度诊断行为问题 → 区分"普适性缺陷"和"项目特有问题" → 产出结构化改进建议
2. **改进审核 Skill**：交叉判断 AK47 设计原则 vs 用户项目诉求 → 五档裁决（接纳/修改/暂缓/拒绝/转上游）→ 给出具体修改方案

这意味着 AK47 的规则体系本身也在被"提案→审核→改进"的流程管理。

### 🚦 Hook 警告映射：不靠 AI 记得

Qoder Hook 在特定操作时触发 Shell 脚本，输出带 🔴/🟡 标记的警告。AK47 预定义了 6 种警告类型和对应的补救路径：

| Hook 类型 | 警告信号 | 强制加载的 Skill | 响应动作 |
|-----------|---------|-----------------|---------|
| TDD 偏离 | 代码文件无对应测试文件 | `ak47-skill-test-driven-development` | 停止编码 → 补测试 → 通过后继续 |
| Artifacts 缺失 | 写代码前 proposal/design/specs/tasks 不齐 | `openspec-propose` 或对应 artifact Skill | 停止编码 → 补齐缺失 artifact → 用户评审通过后继续 |
| Artifacts 全齐 | proposal/design/specs/tasks 全部完成 | `ak47-skill-critical-review` | 停止推进 → 执行 critical-review → 用户批准后 apply |
| 缺少代码审查 | Task 完成/提交前无审查记录 | `ak47-skill-code-review` 或委托 `ak47-agent-reviewer` | 停止提交 → 执行代码审查 → 审查通过后继续 |
| 测试覆盖不足 | 提交前 >30% 源码文件无测试 | `ak47-skill-test-driven-development` | 停止提交 → 补测试 → 覆盖达标后继续 |
| 文档缺失 | 关键文档未更新 | `ak47-skill-critical-review` | 停止提交 → 补文档 → 审查后继续 |

**设计意图**：AI 不用"记得"看到警告该干什么——映射表已经把每种警告绑定了对应的 Skill 和动作序列。Hook 负责检测和告警，规则负责指定补救路径。

### 🔍 状态实时推断：不信任缓存

OpenSpec 工作流的状态（proposing → specing → designing → tasking → implementing → testing → reviewing）**永远基于最新文件内容实时推断**（`always-refresh`），不依赖任何缓存或手动维护的状态字段。

**具体做法**：
- 每次查询 `/status` 都重新扫描 `openspec/changes/` 目录
- 检查每个 Artifact 的文件存在性和内容完整度（proposal.md 是否含 Intent/Scope，specs 是否含 Requirements/Scenario，tasks.md 中 `- [x]` 占比）
- 状态是推断结果，不是存储字段

**避免了什么**：常见做法是用一个 `workflow-state.yaml` 文件手工记录状态，一旦漏更新或并发修改，状态就脏了。实时推断消除了这个不一致窗口。

### ⏰ 阻塞检测：主动提醒而非被动等待

会话开始或状态查询时，自动检查所有 `implementing` 状态的 Change：

- 最后活动时间超过 2 天未更新
- 且任务完成率 < 50%

满足条件时 AI 主动提示阻塞状态，询问是否遇到问题，提供 `/unblock` 建议。

**用户体感**：什么都没做，AI 主动提醒"有个 Change 两天没动了，要不要看看？"——但实际上背后有一套定时检测逻辑在跑。

### 🏗️ 老项目接入：三种路径，不强制迁移

检测到项目有代码但 `openspec/specs/` 为空时，提供三种策略而非强制一次性迁移：

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| 增量接入（推荐） | 新功能走完整 OpenSpec 流程，老功能暂不处理 | 持续迭代中的项目 |
| 基线快照 | 逆向工程为现有代码创建 Spec 基线 | 需要全面梳理的遗留系统 |
| 混合模式 | 已规范模块放 `openspec/specs/`，待规范模块放 `openspec/legacy/` | 分阶段迁移 |

**设计考量**：不因为工具引入而阻断业务开发。老项目可以零成本接入，逐步积累 Spec。

### 💬 反馈行为准则：建设性直率

AK47 约束了 AI 的反馈风格——不是功能特性，而是行为契约：

**分级反馈**：🔴 Critical（必须修复）→ 🟡 Important（建议修复）→ 🟢 Optional（可选优化），每个问题附带位置、证据、建议。

**禁止讨好和对抗**：
- ❌ "你的问题很好""我完全同意"（除非确实如此）
- ❌ "你错了，这样做不行"（没有解释和方案）
- ✅ "这个方案的目标是 X，但实现方式有问题：Y 会导致 Z 风险。建议改用 A 方案，因为 B 证据表明它能避免这个问题。"

**为什么重要**：AI 讨好用户会隐藏问题，AI 对抗用户会破坏协作。建设性直率是二者之间的第三条路——直接但不说教，质疑但有证据，否定但有替代。

### 🚫 禁止行为清单：8 条绝对红线

AGENTS.md 末尾有一条不能商量的清单，每一条背后都有具体教训：

| # | 规则 | 背后的问题 |
|---|------|-----------|
| 1 | 禁止臆想 API/配置 | AI 会"发明"不存在的 API，因为训练数据里见过类似命名 |
| 2 | 禁止过度设计 | AI 倾向于为"以后可能需要"添加抽象层 |
| 3 | 禁止表面修复 | AI 擅长修 symptom 不查 root cause |
| 4 | 禁止跳过 Skills | 对应 1% 规则——AI 永远能找到"不需要"的理由 |
| 5 | 禁止无目标执行 | "修复 bug"不是目标，"写重现测试并让它通过"才是 |
| 6 | 禁止忽略 Hook 警告 | Hook 是最后一道防线，跳过等于防线失效 |
| 7 | 禁止跳过门控 | 每个阶段必须等人确认，跳过需写偏离日志 |
| 8 | 禁止水平切 Spec | Spec 必须端到端用户价值，违反即致命 |

**设计要点**：这些规则不是"建议"，是"禁止"。措辞选择上，不用"应该考虑"而用"禁止"，不给 AI 留模糊空间。

### 📚 知识沉淀自动触发

经验不会自然积累——需要强制触发。AK47 在以下场景自动触发知识沉淀流程：

- 完成工具/框架官网调研后
- 解决 P0/P1 级 Bug 后
- 做出重要技术决策后
- 发现可复用的最佳实践或反模式后

触发后会根据场景判断：当前会话内即时沉淀用主 Agent + `ak47-skill-experience-summarization`，需要跨会话对比历史经验时委托 `ak47-agent-knowledge-engineer` 独立完成。

**为什么是"自动触发"而非"建议触发"**：人天然会忘记记录经验。"记得沉淀一下"等于不会沉淀。把触发条件写进规则，AI 主动发起。

## 对比其他方案

| 设计点 | 常见做法 | AK47 的做法 | 差异 |
|--------|---------|-----------|------|
| 流程控制 | Prompt 里写"请先写测试" | 1% 规则 + 跳过判定 + Hook 警告 | 软约束 → 硬约束 |
| 阶段确认 | AI 自动判断"可以继续了" | 11 个门控点 + 确认格式规范 | AI 自主 → 人等 AI |
| Spec 拆分 | 按模块/包/层次拆 | 强制垂直切片 + 审核检查 | 互相依赖 → 可独立交付 |
| 规则跳过 | 没有记录 | 偏离日志 + 四步批准 | 不可追溯 → 可追溯 |
| 规则执行 | 全靠 AI 读 Prompt | Hook + AGENTS.md + AI 自律三层 | 单层 → 多层 |
| 自我进化 | 人工改规则 | 改进提案+审核双 Skill | 外部 → 内置 |
| 状态管理 | YAML 文件手工记录 | 文件内容实时推断（always-refresh） | 易脏 → 永真 |
| 经验沉淀 | "记得记录一下" | 场景自动触发 + Agent 分工分流 | 靠人记 → 系统推 |
| AI 反馈风格 | 讨好或对抗 | 建设性直率（分级 + 证据 + 替代方案） | 情绪化 → 工程化 |
| 旧项目接入 | 强制一次性迁移 | 增量/基线/混合三策略 | 阻断业务 → 零成本渐进 |
