# 04 · Agent 分工决策方法论

> **解决什么问题**:多 AI Agent 协作时,主 Agent 和子 Agent 如何分工?避免"教条式职责分工"和"过度使用子 Agent"两个极端  
> **预计阅读**:8 分钟  

---

## 背景

在多 AI Agent 协作体系中,一个核心问题是:什么时候该由主 Agent 自己干,什么时候该委托给子 Agent?

错误答案 A:产品需求是 Product Agent 的职责,所以必须用子 Agent。(教条式职责分工)
错误答案 B:为了让 Agent 各司其职,每个小任务都创建一个子 Agent。(过度使用子 Agent)
正确答案 C:权衡上下文连续性和独立判断力——需要记忆的用主 Agent,需要客观立场的用子 Agent。

---

## 来源

如果按"专业领域"分配 Agent:

```
产品需求 → Product Agent
架构设计 → Architect Agent
代码编写 → Developer Agent   
代码审查 → Reviewer Agent
```

可能就会发现两个问题:

**问题 1:子 Agent 缺少全面的对话历史。** 用户和主 Agent 讨论了 10 分钟需求背景,但委托给 Product Agent 写需求文档时,子 Agent 对背景的了解依赖主Agent的输入,产出的文档和用户预期存在一定比例的信息损失——返工率风险增加。

**问题 2:子 Agent 泛滥。** 一个简单功能的开发,创建了 Product Agent → Architect Agent → Developer Agent → Reviewer Agent 四个子 Agent,每个都要依赖主Agent整理信息作为输入,重新理解项目上下文。总耗时反而比主 Agent 自己做完更长。

### 核心洞察

**子 Agent 的价值不在"职责分工"，而在"效率加速"和"独立判断"。**

| 子 Agent 的定位 | 正确定位 ✅ | 错误定位 ❌ |
|--------------|-----------|-----------|
| 核心价值 | 并发执行、缩短总耗时 | 按专业领域划分职责 |
| 适用时机 | 任务可并行时提示"可以并发" | 因为"这个任务属于某 Agent" |
| 典型场景 | 3 个 Spec → 3 个子 Agent 并行编写 | 写需求 → Product Agent（即使主 Agent 有完整上下文） |

---

## 原理

### 决策矩阵

| 场景特征 | 推荐方案 | 原因 |
|---------|---------|------|
| 需要上下文连续性 | 主 Agent + Skill | 主 Agent 拥有完整对话历史，能基于前文做出连贯决策 |
| 需要独立判断力 | 子 Agent + Skill | 子 Agent 无历史包袱，能提供客观、不受干扰的判断 |
| 可以并发执行 | 多个子 Agent + Skill | 子 Agent 可并行运行，显著提升效率 |

### 主 Agent 负责的场景

**判断标准：如果任务需要"记得之前说过什么"，用主 Agent。**

| 场景 | 为什么需要上下文连续性 |
|------|---------------------|
| 产品需求讨论与编写 | 需要基于对话历史理解需求演变，多轮交互澄清模糊点 |
| 代码设计与编写 | 需要理解项目整体架构和已有代码，保持风格一致性 |
| 经验沉淀（即时场景） | 从当前对话中提炼经验，需完整上下文才能准确总结 |
| 架构决策 | 需要权衡多个方案的历史背景和讨论过的约束条件 |

### 子 Agent 负责的场景

**判断标准：如果任务需要"不受历史影响的独立判断"或"可以并行加速"，用子 Agent。**

| 场景 | 为什么适合子 Agent |
|------|-------------------|
| 审查类任务（Code Review、Spec 评审） | 需要独立视角，避免"作者审查自己代码"的盲区 |
| 并发执行任务（多 Spec 并行、多 Task 编码） | 子 Agent 是效率工具，并发显著缩短总耗时 |
| 专业领域深度分析（安全审计、性能分析） | 专业 Agent 有独立的系统提示和领域知识 |
| 跨会话/跨项目的经验整理 | 需要独立视角，不受单一会话上下文限制 |

---

## 核心方法

### 决策流程

```
任务到来
  │
  ├─ 需要上下文连续性？ ── YES ──→ 主 Agent + Skill
  │                                    │
  │                                   NO
  │                                    │
  ├─ 需要独立判断力？ ── YES ──→ 子 Agent + Skill
  │                                    │
  │                                   NO
  │                                    │
  ├─ 可以并发执行？ ── YES ──→ 多个子 Agent + Skill（并行）
  │                              │
  │                             NO
  │                              │
  ├─ 需要专业领域知识？ ── YES ──→ 专业子 Agent + Skill
  │                                    │
  │                                   NO
  │                                    │
  └─ 默认 ──→ 主 Agent + Skill
```

### 委托话术规范

```
✅ 正确：「检测到 5 个 Task 互不依赖，可以分配给 3 个 Code Agent 并行开发。
        预计耗时从 15 分钟缩短到 5 分钟。是否开始？」

✅ 正确：「Spec 已评审通过，接下来可以并发生成 3 个模块的 Spec，
        预计耗时从 15 分钟缩短到 5 分钟。是否启动并发流程？」

❌ 错误：「根据职责分工，产品需求应由 Product Agent 编写。」
       （忽略了主 Agent 的上下文优势）
```

### 检查清单

任务分配前自查：

- [ ] 该任务是否需要上下文连续性？（是 → 主 Agent）
- [ ] 该任务是否需要独立判断力？（是 → 子 Agent）
- [ ] 该任务是否可以并发执行？（是 → 多个子 Agent）
- [ ] 该任务是否需要专业领域知识？（是 → 专业子 Agent）
- [ ] 主 Agent 是否在可并发场景主动提示了用户？

## 对比其他方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **上下文连续性优先（本方案）** | 主 Agent 有完整对话历史，产出连贯一致，减少返工 | 无法并发、主 Agent 负担重 | 需要多轮交互的复杂任务 |
| **教条式职责分工** | 分工明确，角色边界清晰 | 子 Agent 缺乏上下文，产出质量下降，返工率高 | 不推荐 |
| **全主线（完全不用子 Agent）** | 上下文完整、零协调成本 | 无法并发、无法获得独立审查视角 | 简单项目、个人开发 |
| **全子 Agent（主 Agent 只做路由）** | 高度并行、职责清晰 | 上下文碎片化、协调成本高、全局把控弱 | 不推荐 |

### 三种常见错误

| 错误 | 表现 | 后果 | 修正 |
|------|------|------|------|
| **教条式职责分工** | 严格按"产品类型用 Product Agent、代码用 Code Agent"分工 | 子 Agent 缺乏历史,频繁返工 | 优先判断是否需要上下文 |
| **过度使用子 Agent** | 每个小任务都创建子 Agent | 主 Agent 失去全局把控，协调成本高 | 仅在有并发收益或需要独立判断时使用 |
| **忽略并发提示** | 按顺序执行可并行的任务 | 总耗时过长，用户体验差 | 检测到可并发任务时主动提示 |
