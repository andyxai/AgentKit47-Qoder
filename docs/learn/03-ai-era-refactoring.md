# 03 · AI 时代的代码重构：从"能跑"到"更好维护"

> **解决什么问题**：AI 生成代码快但质量参差不齐——函数过长、重复代码、条件嵌套、贫血模型  
> **预计阅读**：12 分钟  
> **原始经验**：[BP-refactoring-improving-existing-code.md](../../.ak47/experiences/best-practices/BP-refactoring-improving-existing-code.md)
>
> 📎 知识来源：[Martin Fowler《Refactoring》](https://refactoring.com/) · [Refactoring Guru](https://refactoring.guru/)（可视化重构手法速查）

---

## 背景

AI 编程工具让代码生成效率提升了很多，但也带来了新问题：AI 倾向于一个函数写完所有逻辑、在不同地方生成相似的代码、不考虑长期维护。重构是把 AI 生成的"能跑的代码"变成"可维护的好代码"的核心手段。

---

## 来源

方法论源自 Martin Fowler 的《重构：改善既有代码的设计》（[refactoring.com](https://refactoring.com/)），针对 AI 生成代码的特点做了适配。核心发现：AI 生成代码最常见的六类问题——代码臃肿、数据结构混乱、耦合过度、条件逻辑复杂、接口设计缺陷、继承滥用——每一类都有标准化的重构手法。

---

## 原理

### AI 生成代码的典型问题

| 问题 | AI 为什么这么写 | 后果 |
|------|---------------|------|
| **过长函数** | 倾向"一个函数完成全部逻辑" | 可读性差、难以复用、修改易引入 Bug |
| **重复代码** | 不同 prompt 生成的代码互不知晓 | 改一处漏一处、代码膨胀 |
| **贫血模型** | 只生成 get/set，不封装行为 | 业务逻辑散落各处、数据不设防 |
| **箭头型代码** | 多层 if-else 嵌套 | 可读性极差、测试需覆盖大量组合 |
| **魔法数** | 不会主动提取常量 | 含义不明、修改困难 |
| **过度设计** | 倾向于加"未来可能用到"的抽象 | 代码结构复杂、实际用不到 |

### 重构铁律（不可突破）

| 铁律 | 含义 |
|------|------|
| **安全网优先** | 重构前必须有自动化测试套件 |
| **小步快跑** | 每次只做一个微小重构，立刻运行测试 |
| **两顶帽子**（Kent Beck） | 重构时不改功能，改功能时不重构——你始终清楚自己戴着哪顶帽子 |
| **三次法则**（Don Roberts） | 同一逻辑第一次直接写，第二次可容忍，第三次必须重构（"Three strikes and you refactor"） |

## 核心方法：六大类坏味道与重构手法

### 第一类：代码臃肿

| 坏味道 | 核心手法 | 一句话 |
|--------|---------|--------|
| 重复代码 | **提炼函数** | 同一段代码出现三次，提取为独立函数 |
| 过长函数 | **提炼函数** | 需要注释说明用途的代码块，都提炼为函数 |
| 过大类 | **提炼类** | 一组相关字段/函数搬到新类 |
| 无用类 | **内联类** | 类没足够职责，合并到调用方 |

### 第二类：数据与结构问题

| 坏味道 | 核心手法 | 一句话 |
|--------|---------|--------|
| 基本类型偏执 | **以类取代类型码** | 用 string 表示手机号 → 封装为 PhoneNumber 类 |
| 数据泥团 | **引入参数对象** | startDate + endDate 总是一起出现 → 封装为 DateRange |
| 过长参数列 | **引入参数对象** | 参数 > 3 个 → 封装为参数对象 |
| 贫血模型 | **搬移函数** | 把操作该类字段的逻辑搬进类内部 |

### 第三类：耦合过度

| 坏味道 | 核心手法 | 一句话 |
|--------|---------|--------|
| 依恋情结 | **搬移函数** | 函数大量调用另一个类的 get → 搬过去 |
| 消息链 | **隐藏委托关系** | `a.getB().getC().doX()` → 在 A 上加 `doX()` |
| 中间人 | **移除中间人** | 类只是委托不做实事 → 删除它 |

### 第四类：条件逻辑复杂

| 坏味道 | 核心手法 | 一句话 |
|--------|---------|--------|
| 过长 switch/if-else | **以多态取代条件表达式** | 每个分支变成子类的一个方法 |
| 箭头型嵌套 | **以卫语句取代嵌套条件** | 异常情况先 return，主逻辑放最后平级 |
| 魔法数 | **以字面常量取代魔法数** | Python: `if status == OrderStatus.CANCELLED:` / Java: `if (status == OrderStatus.CANCELLED)`（替换 `if status == 3` 这类魔法数） |

### 第五类：接口设计缺陷

| 坏味道 | 核心手法 | 一句话 |
|--------|---------|--------|
| 函数名不清晰 | **函数改名** | `handleData()` → `parseUserProfile()` |
| 查询改混合 | **查询与修改分离** | `getUser()` 里更新登录时间 → 拆成两个函数 |

### 第六类：继承体系混乱

| 坏味道 | 核心手法 | 一句话 |
|--------|---------|--------|
| 拒绝遗赠 | **以委托取代继承** | 子类只用到超类 20% 的方法 → 改为持有超类实例 |
| 继承滥用 | **以委托取代继承** | User 类继承 ArrayList → 改为持有 List 字段 |

## 对比其他方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **持续小重构（推荐）** | 风险低、可融入日常开发 | 需要持续投入 | AI 辅助开发的日常标配 |
| **大扫除式重构** | 一次性解决大量问题 | 风险高、易被推迟 | 严重腐化的遗留系统 |
| **AI 生成 + 不重构** | 交付快 | 代码质量持续下降、维护成本指数上升 | 一次性脚本 / 原型 |
| **AI prompt 前置约束** | 源头减少坏味道 | 不能完全杜绝、需要持续优化 prompt | 配合重构使用 |

---

## AI 时代的最佳实践

1. **AI 生成 → Code Review → 重构 → 测试 → 上线**：把重构作为 AI 开发的必备环节
2. **提示词前置约束**：在 prompt 中要求"函数不超过 30 行"、"提取常量为枚举"等
3. **善用自动化重构工具**：IDE 的 Extract Method、Move Method 一键完成，安全高效
4. **持续小重构，而非一次性大扫除**：把重构融入每次代码修改，防止技术债累积

## 重构前安全检查

- [ ] 测试套件是否覆盖目标代码？
- [ ] 测试是否全部通过？
- [ ] 是否在独立分支上操作？
- [ ] 是否已提交当前可用版本？
