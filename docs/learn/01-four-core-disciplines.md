## 一、TDD：小步快跑的开发纪律

### 来源

Kent Beck 在 1990 年代提出。核心洞察：**先写测试会倒逼你思考"这段代码到底要做什么"，从而写出更简洁、更可测试的设计**。不是"为了测试而测试"，而是"用测试驱动设计"。

### 原理

| 问题 | TDD 怎么解决 |
|------|------------|
| 代码写完不知道怎么验证 | 测试就是可执行的验收标准 |
| 改一个地方牵动全身 | 有测试兜底，改完跑一遍就知道坏没坏 |
| 新人看不懂代码 | 测试用例比注释更可靠——它展示了代码的预期行为 |
| 一写就多，过度设计 | 只写让测试通过的代码，自然杜绝 YAGNI 违例 |

### 核心方法：红 → 绿 → 重构

| 阶段 | 动作 | 铁律 |
|------|------|------|
| 🔴 **Red** | 先写一个会失败的测试 | 不许写实现代码 |
| 🟢 **Green** | 写最少的代码让测试通过 | 可以硬编码，不许扩展功能 |
| 🔵 **Refactor** | 优化代码，不改行为 | 不许改业务逻辑，不许加新功能 |

### 对比其他方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **TDD** | 设计导向、安全网、测试即文档 | 学习曲线陡、初始速度慢 | 核心业务逻辑、复杂算法 |
| **先写代码后补测试** | 初始速度快 | 测试往往覆盖不全、代码可能不可测 | 快速原型 |
| **只写测试不重构** | 有安全网 | 代码质量不提升、技术债累积 | 不建议 |

### 落地检查清单

- [ ] 是否先写测试再写实现？
- [ ] 测试是否先失败（验证测试有效）？
- [ ] 实现代码是否为最小量？
- [ ] 重构后测试是否全部通过？

## 二、DDD：复杂业务的建模纪律

### 来源

Eric Evans 2003 年出版《领域驱动设计》。核心洞察：**大多数软件失败不是因为技术不行，而是业务和技术脱节**——产品说的话和代码里的名字对不上。

### 原理

| 问题 | DDD 怎么解决 |
|------|------------|
| 产品和开发各说各话 | 通用语言（Ubiquitous Language）统一术语 |
| 一个模块改了影响全局 | 限界上下文（Bounded Context）按业务边界拆分 |
| 业务逻辑散落在各处 | 领域层集中业务规则，形成充血模型 |
| 微服务拆得太碎或太紧 | 限界上下文是微服务划分的天然依据 |

### 核心方法

| 概念 | 一句话 | 示例 |
|------|--------|------|
| **通用语言** | 所有人用同一套术语 | "订单"不在代码里叫 "Order" 又叫 "TradeInfo" |
| **限界上下文** | 按业务职责拆分独立子域 | 商品 / 订单 / 支付 / 物流 各自独立 |
| **聚合** | 一组关联对象，聚合根是唯一入口 | 订单聚合根管理订单项、地址、支付信息 |
| **领域服务** | 处理跨聚合的业务逻辑 | 支付需同时操作订单和库存 |

### 对比其他方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **DDD** | 业务对齐、边界清晰、可维护性高 | 学习成本高、简单系统过度设计 | 复杂业务系统（电商/金融/物流） |
| **贫血模型（传统三层）** | 简单直接、上手快 | 业务逻辑散落、系统大了就乱 | 简单 CRUD 系统 |
| **CQRS/Event Sourcing** | 读写分离、事件溯源 | 复杂度极高 | 高性能、高审计需求场景 |

### 落地检查清单

- [ ] 项目是否建立了通用语言（术语统一）？
- [ ] 是否按限界上下文拆分了业务子域？
- [ ] 领域模型是否充血（业务逻辑在领域层）？
- [ ] 聚合根是否正确定义？

## 三、调试 SOP：高效排障的标准纪律

### 来源

我整合了 mattpocock 的调试方法论（[mattpocock/skills](https://github.com/mattpocock/skills)）和 AK47 的架构诊断能力。核心洞察：**大多数调试时间是浪费在"试来试去"上，而不是"推理和验证"上**。把老工程师的排查经验标准化，新人也能高效排障。

### 原理

传统调试：凭经验猜 → 改代码 → 试 → 不行 → 再猜 → 循环  
SOP 调试（源自 mattpocock）：构建反馈循环 → 复现 → 最小化 → 假设 → 验证 → 修复 → 复盘

关键差异在于：**假设必须可验证、可证伪**。"可能是代码有问题"是不可验证的坏假设；"当用户 ID 为空时会空指针异常"是可验证的好假设。

### 核心方法：七步法

| 步骤 | 做什么 | 禁止什么 |
|------|--------|---------|
| **1. 构建反馈循环** | 建立快速验证修复是否有效的手段（测试/curl/脚本） | 无反馈循环禁止改代码 |
| **2. 稳定复现** | 找到稳定复现步骤（100% 或尽可能高） | 不可复现的问题禁止进入下一步 |
| **3. 最小化范围** | 二分定位、去掉无关变量，把问题缩小到最小范围 | 不许一次改多个地方 |
| **4. 形成假设** | 3-5 个可证伪的假设，按概率排序 | 不许提笼统假设（"可能是代码有问题"） |
| **5. 仪器化验证** | 一次只改一个变量，用调试器或唯一前缀日志 | 不许同时验证多个假设 |
| **6. 修复闭环** | 先写回归测试（确认失败）→ 写修复代码 → 测试通过 | 不许修完不写回归测试 |
| **7. 清理复盘** | 清除调试日志、commit message 记录根因 | 不许留下调试代码 |

### 对比其他方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **调试 SOP** | 系统化、可复制、效率高 | 需要自律、初期觉得"慢" | 所有 Bug 排查 |
| **凭经验猜** | 经验丰富时可能很快 | 不可复制、容易引入新问题 | 不推荐 |
| **print 大法** | 简单直接 | 效率低、信息杂乱、容易漏关键点 | 简单问题快速定位 |
| **专业调试器** | 精确、强大 | 学习成本高、某些环境不可用 | 复杂逻辑深度调试 |

## 四、反熵增节奏：系统长期健康的治理纪律

### 来源

热力学第二定律告诉我们：封闭系统会自然趋向混乱（熵增）。软件系统也一样——**如果不持续输入秩序，代码会不可避免地腐化**。这不是"开发不努力"，而是自然规律。

### 原理

| 熵增表现 | 反熵增对策 |
|---------|-----------|
| 重复代码越来越多 | 日常小步重构（童子军规则，源自 Robert C. Martin《Clean Code》） |
| 技术债越积越多 | 每周/每月固定时间清理 |
| 架构边界模糊 | 架构守护者持续监控 |
| Bug 修复引入新 Bug | 自动化测试 + CI/CD |
| 人员流失导致知识断层 | 文档持续更新 + 新人培训 |

### 核心方法：五层防护

| 层级 | 节奏 | 手段 |
|------|------|------|
| **日常重构** | 每次修改都优化一处 | 童子军规则（Robert C. Martin）：离开时比来时更干净 |
| **定期清债** | 每周/每月固定时间 | 技术债看板 + 按利息分级 |
| **架构守护** | 持续监控 + 定期评审 | ESLint + 依赖检查 + 熵值度量 |
| **自动化防护** | 每次提交自动运行 | 测试 + CI/CD + Code Review |
| **知识传承** | 持续 | 文档同步更新 + 复盘 + 技术分享 |

### 对比其他方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **反熵增节奏（持续治理）** | 系统长期健康、维护成本低 | 需要持续投入、短期"看不到产出" | 所有长期维护的项目 |
| **大扫除式重构** | 一次性解决大量问题 | 风险高、容易被推迟、永远排不上 | 严重腐化的遗留系统 |
| **不治理** | 短期交付快 | 系统必然走向不可维护 | 一次性原型/脚本 |

## 落地路线图

从易到难，逐步推进：

| 阶段 | 推行内容 | 预期效果 | 周期 |
|------|---------|---------|------|
| **一** | 调试 SOP + 代码规范 | 快速提升排障效率和代码质量 | 1-2 周 |
| **二** | TDD（核心业务模块试点） | 养成小步迭代、测试先行的习惯 | 1-2 月 |
| **三** | DDD（新系统或重大重构） | 建立业务模型和架构规范 | 2-4 月 |
| **四** | 反熵增节奏 | 保障系统长期健康 | 持续 |
