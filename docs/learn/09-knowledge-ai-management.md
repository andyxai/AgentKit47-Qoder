# 项目知识资产的 AI 化管理

---

## 背景

AI 编程助手能否高效工作，很大程度上取决于它知道什么。一个项目里的知识——领域术语、架构约束、踩过的坑、最佳实践——如果散落在各处或写得太长太乱，AI 要么找不到，要么当噪音直接忽略。

这一篇整合了 AK47 在知识资产管理上的四个核心方法：分层加载、术语归一化、Rules 工程化、经验沉淀。

---

## 来源

### 踩过的坑

最初的规则体系是一份 172 行的 hard-rules.md，什么都要管。结果 AI 在写 CSS 的时候被塞了一堆数据库规范，真正的安全红线反而被淹没。

术语管理更是打过两场仗：先试了双轨制（人看 glossary.md + AI 看 CONTEXT.md），两份文档不同步，两边都不准。经验沉淀也经历过——坑记了一堆，但文件散落各处，AI 根本找不到。

### 最终收敛的方向

奥卡姆剃刀原则贯穿始终：能用一份文档解决的不要两份，AI 不需要的不注入上下文，不重要的不要单独成篇。

## 核心方法

### 1. 分层加载策略

项目知识分两级，按加载时机区分：

```
┌─────────────────────────────────┐
│  宪法层 — 全量加载（每次会话必须读） │
│  constitution.md + hard-rules.md │
│  内容：不可妥协的底线、架构原则      │
│  特点：短而精，< 200 行            │
└─────────────────────────────────┘
               ↓ 约束
┌─────────────────────────────────┐
│  经验层 — 按需检索（相关性触发）     │
│  experiences/ + CONTEXT.md       │
│  内容：最佳实践、领域术语、历史决策   │
│  特点：分文件存储，路径过滤精准注入  │
└─────────────────────────────────┘
```

**核心原则**：宪法层每次必读，经验层按需加载。不重要的经验不要单独成文件——合并在一个相关主题文档里。

**对应关系**：这个两层设计映射到[人机协同三层分离模型](../design/decisions/09-intent-process-execution-separation.md)中的宪法层和流程层。

### 2. 术语归一化：CONTEXT.md

来自 mattpocock/skills 的设计，核心思想是**一份结构化 Markdown 同时服务人和 AI**：

```markdown
**Customer（客户）**:
A person or organization that places Orders.
_Avoid_: Client, buyer, account, user
```

三个关键决策：

| 决策 | 选了 | 理由 |
|------|------|------|
| 文档数量 | 一份 CONTEXT.md | 奥卡姆剃刀：不需要人一份 AI 一份 |
| 创建时机 | 懒加载（第一次遇到术语时） | 预生成空模板反而让用户误以为"已配置" |
| 禁止词 | Avoid 列表 | 告诉 AI"不能用什么"比"还可以叫什么"更有效 |

AI 在对话中遇到 Avoid 列表中的词时自动校正，无需人工干预。

### 3. Rules 工程化：用路径过滤精准注入

**问题**：规则太多 → AI 无视；没有路径过滤 → CSS 开发时加载数据库规则 → 上下文污染。

**解法**：按领域拆分 + 路径过滤，让 AI 写什么代码看什么规则：

| 规则文件 | 适用路径 | 触发时机 |
|---------|---------|---------|
| `cli-rules.md` | `src/cli/**/*` | 处理 CLI 代码 |
| `core-architecture-rules.md` | `src/core/**/*` | 处理核心模块 |
| `test-rules.md` | `tests/**/*` | 编写测试 |

**两条底线**：
- 单文件 < 200 行（来自 Claude Code 官方建议："长而不专注的规则会被忽略"）
- Rules 定义"不能做什么"（底线），Skills 定义"如何做"（方法），两者不混淆

### 4. 经验沉淀：什么时候记、记多细

**不需要单独成篇的**，合并到已有专题文档中。只有满足以下条件才沉淀为新经验：

1. 踩过的坑**可能再次发生**（否则不值得记）
2. 有**可操作的避免步骤**（否则记了也没用）
3. 属于某个**已有分类**（否则不知道放哪）

**分类指南**（简化版）：

| 类型 | 放哪里 | 何时加载 |
|------|-------|---------|
| 硬约束（不能做的） | 宪法层 rules/ | 每次 |
| 最佳实践（这样做更好）+ 实践记录（那样做会有问题） | 经验层 practices/ | 场景匹配时 |
| 架构决策（为什么选这个） | 经验层 decisions/ | 相关变更时 |

**核心原则**：best-practices 和 pitfall-records 合并——"这样做是对的"和"那样做会出事"对比着写才有效果。

---

## 原理

### 人类阅读心理学同样适用于 AI

Claude Code 官方建议单文件 < 200 行，因为"长而不专注的规则会被忽略"。AI 上下文窗口虽大，但注意力有限——超过一定长度的内容，关键信息淹没在噪声中。

### 分层加载减少了上下文污染

宪法层短而精，经验层按需检索。每次会话的上下文空间留给真正需要的知识，而不是被"万一用得上"的历史经验占满。

### 奥卡姆剃刀降低了维护成本

一份文档 > 两份（术语不用双轨），合并 > 拆分（经验不用过度分类），不重要的不单独成篇。维护成本随文档数量非线性增长。

---

## 要点

1. 分层加载是核心：宪法层全量（每次必读），经验层按需（场景触发）
2. 路径过滤是引擎：让 AI 写什么代码看什么规则，不相关的不注入
3. Avoid 列表 > 别名列表：告诉 AI 不能用什么比还可以叫什么直接
4. 懒加载 > 预生成：空文档比没有文档更糟，它让人误以为已配置
5. 经验不值得过度分类：best-practices + pitfall-records 合一起写，对比着来
