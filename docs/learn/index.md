# 📚 AK47 学习中心

> ⚠️ 以下内容基于个人项目实践和学习的总结，难免有错漏。个人项目中的需求、设计、代码等由国内 LLM 辅助完成（以下"我+AI"指代这种协作方式）。这本身也是一次探索——在 LLM 能力仍有边界的前提下，怎么通过工程化手段提升产出质量和协作效率。

---

## 项目背景

AK47 并非凭空诞生，而是**站在多个开源项目的肩膀上整合而成**。以下是实际研读过的 5 个核心参考项目，各有取舍：

| 开源项目 | 学习到的核心思路 | AK47 的取舍 |
|---------|-----------------|-----------|
| **[spec-kit](https://github.com/github/spec-kit)** | 规范驱动开发（SDD）、宪法层文档体系、`/speckit.*` 命令链、Community Extensions 生态 | 框架偏重，扩展机制基于独立插件市场（非事件驱动）——取其「Spec 层级」理念，扩展层用自己的 Agent/Skill + Hooks |
| **[OpenSpec](https://github.com/Fission-AI/OpenSpec)** | 渐进式 Spec 管理、Delta 归档机制、`/opsx:*` 命令链 | 偏文档规范，缺少 Agent/Skill 编排层——作为底层 Spec 引擎，AK47 在上层补充能力层（L1/L2/L3 分级为 AK47 自身扩展） |
| **[mattpocock/skills](https://github.com/mattpocock/skills)** | `/diagnose` 六步调试（AK47 扩展为七步，增加清理+沉淀）、`/grill` 一次一问盘问、TDD 垂直切片（Tracer Bullet）、深层模块（Deep Module） | 面向个人 Claude Code，无团队配置管理——取其方法论精华，融入 AK47 的 Skill 体系 |
| **[Superpowers](https://github.com/obra/superpowers)** | TDD/计划编写/代码审查/子 Agent 驱动的 AI 编码方法论体系 | 流程编排偏线性（强制单向流转）、灵活性不足、无知识沉淀——取其 Skill 骨架，AK47 补充编排和沉淀层 |
| **[gstack](https://github.com/garrytan/gstack)** | 虚拟工程团队（23 个专家角色）、Think→Plan→Build→Review→Test→Ship→Reflect 七阶段、CEO 级需求盘问（`/office-hours`） | 功能完整的开源方案之一——取其角色化 Agent 思想、多 Agent 协同模式和需求盘问方法论 |

> 📂 除上述 5 个核心项目外，还浏览过 agent-skills、karpathy-skills、anthropics-skills、compound-engineering-plugin 等项目，仅作参考，未纳入 AK47 体系。

**核心思路**：学习上述所有项目的优秀思想，基于自己的实际需求，给自己打造一个**适合自己的 AI 开发环境配置工具**。经过多次迭代，发现这套配置体系对其他人也有参考价值，于是尝试调整以支持在多个项目中使用。

> 但无论框架如何演进，扩展层始终会保留这些最适合自己的 Agent 和 Skill。

## 思想演进：从轻量到体系化

### 第一阶段：实用主义起步（"先跑起来再说"）

最初只是自己在用 Qoder 和 Claude Code，搭配几个 GitHub 上评分较高的开源 Skills。目标很简单：**让 AI 少犯低级错误，产出能直接用**。看到 mattpocock 的 `/diagnose` 七步调试法、Superpowers 的 TDD 流程、gstack 的角色化 Agent 思路，第一反应是"这个好，拿过来改改就能用"。

这个阶段的核心特征是**轻量、灵活、个人化**——Skills 是独立的 Markdown 文件，改一个不影响其他的；流程靠 AGENTS.md 里的自然语言描述，AI 理解多少算多少；没有严格的阶段门控，靠人判断"差不多了就下一步"。

### 第二阶段：意识到"万变不离其宗"

随着各种 AI 开发范式层出不穷——SDD、BDD、TDD 的 AI 变体、Harness 设计、Agent 编排——逐渐看清一个事实：**底层逻辑上，这全是传统软件工程思想的复兴**。

`/speckit.constitution` 本质是在做架构约束管理，《代码大全》里早就讲过。Harness 七层架构就是对分层设计模式的工程化落地。Phase Gate 门禁检查，不过是传统的质量门（Quality Gate）换了个 AI 友好的外壳。TDD 的红-绿-重构循环，在 AI 时代无非是把"人写测试"换成了"AI 先写测试，人做确认"。

于是回头重读《代码大全》《重构》《领域驱动设计》等经典，确认了一个判断：**大部分所谓"AI 原生开发方法论"，思想根系都扎在传统软件工程里。AI 改变的是执行效率，不是工程原理**。

### 第三阶段：从"够用就行"到"需要体系化"

当 Skills 从 5 个涨到 30 个，Agent 从 3 个涨到 9 个，四个问题逐渐浮现：

- **流程断裂**：TDD Skill 知道要先写测试，但不知道什么时候该触发 Code Review Skill
- **规则冲突**：AGENTS.md 越写越长，不同规则之间开始互相矛盾
- **维护噩梦**：每次升级都要手动检查 30 多个 Skill 文件、9 个 Agent 配置、多条规则是否一致
- **新人无法独立使用**：这套配置隐含了大量"我知道但没写下来"的约定

这时候才看清一个事实：**轻量化的代价是隐性知识太多。自己能用，换个人就用不了**。

### 第四阶段：接受"重度化不可避免"

最初觉得 spec-kit 太重——constitution、plan/spec/tasks 多阶段、extension/preset 双层定制。但自己的方法论也在变重，回头再看，发现 spec-kit 那些"重"设计恰好对症：

| 之前的问题 | 重度化怎么解 |
|-----------|------------|
| 流程靠 AGENTS.md 散文描述，AI 容易遗漏 | 命令模板内嵌 Phase Gate + Handoff，不靠 AI 自己"记得" |
| 规则分散、互相矛盾 | Constitution 作为唯一真源，所有阶段强制检查 |
| 升级时手动比对 30+ 文件 | Extension/Preset 分层，核心与定制解耦 |
| 隐性知识无法传递 | 结构化模板 + Checklist 自检，把"我知道"变成"系统知道" |

**重度化的本质不是"变复杂"，而是"把隐性知识显性化、把软约束硬化"**。这对多角色协作的大项目是刚需，但对小项目或独立开发者确实可能用力过猛。

### 第五阶段（当前）：探索"框架 + 插件"的最优解

接受重度化之后，核心问题变了：**以个人精力，能不能持续维护一个越来越重的体系？**

自己开发确实灵活，但随着体系膨胀，维护成本非线性增长。顺着这个方向思考：

- 是不是应该把 AK47 定位成 spec-kit 这类框架的**扩展插件层**？
- 框架管"规范流程"（宪法、Spec、Plan、Tasks），插件管"能力注入"（Agent、Skill、经验库）
- 框架升级由社区驱动，自己只维护差异化的能力和经验

这个思路还在验证。开源框架的灵活性能否承载所有定制需求？插件机制的解耦程度是否够用？都需要继续学习和实践才能下判断。

**核心矛盾**：个人开发者的灵活性与长期可持续性之间的张力。AK47 当前选择了一条务实路径——"保持独立的前提下，尽可能与成熟框架对齐接口"，既不锁定于某一框架，也不重复造轮子。

---

### 关于框架之争

各类 AI 开发框架各有长短，选择本质上是个适配问题，不存在普适的"最优解"。当前这些框架解决的仍是**规范化和辅助层面的问题**——帮你把需求写清楚、把 Spec 拆合理、把流程串起来、把规则落下去。"AI 端到端完整交付一个可上线产品"是另一个命题，远不止这些框架在做的事情。

如果某位读者已经有了自己顺手的方法和工具，且在实践中证明有效——那你的方案就是对的，适合你的就是最好的。这里的总结只是一份个人实践笔记：一个独立开发者从"轻量够用"走到"体系化思考"的真实路径。我学习尚浅，还需要更丰富的实践来打磨判断力。

> 📌 **当前态度**：不对任何一种方案做终局判断。不会因为研究了某个框架就强力站队推销它。继续学、继续练、继续总结。如果哪天发现一个足够灵活的框架能承载所有需求，那就切过去；如果发现还是自己维护更合适，那就接着干。重要的是**保持思考和记录，不盲从不教条**。

## 📖 阅读建议
